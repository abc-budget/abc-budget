# Story 4.4.1 — Override triplet key — PLAN (consistency fix)

> Straight-to-implementation, test-first (no design gate). Branch `story/4.4.1-override-triplet-key` off
> `main` @ `71623e4`. Captured in Q-012 + ENT-014.

## Why
The persisted override map (4.4) keys on `hash` ALONE, but the footprint PK is the triplet `(hash, year,
month)` (ENT-014). An override targets a specific footprint → it must key by the SAME triplet (month-
narrowing: a hash-collision can only mis-apply WITHIN the same month, matching the dedup guarantee).
Probability impact ~10⁻⁶⁹ — this is for CONSISTENCY/correctness, not a live bug.

## The change — keyed composite `${hash}|${year}|${month}` (the override map stays `Map<string,string>`)
**Shared derivation (the byte-match guarantee):** extract `footprintYearMonth(date): { year, month }` from
`footprint/derive-footprint.ts` (UTC: `getUTCFullYear()`, `getUTCMonth()+1`); `deriveFootprint` uses it AND
the override key uses it → a row and its footprint compute the SAME (year,month) by construction.

**Key helpers** (in `categorize-with-overrides.ts`, exported so 4.7 imports them):
- `overrideKey(hash, year, month): string` = `` `${hash}|${year}|${month}` ``.
- `overrideKeyForRow(row): string` = `overrideKey(row.hash, ...footprintYearMonth(row.date))`.

**The FOUR production sites** (all must move together or they desync):
1. `loadOverrideMap` (4.4) — `overrideMap.set(overrideKey(fp.hash, fp.year, fp.month), fp.categoryId)`.
2. `resolveCategory` L2 (4.4) — `ctx.overrideMap.get(overrideKeyForRow(row))`.
3. `resetToRules` (4.4) — `ctx.overrideMap.delete(overrideKeyForRow(row))`. ← the 4th site (not in the
   prompt's list, but it keys by `row.hash` → MUST move or reset desyncs).
4. `autoCategorize` (4.7) — `isManual = (row.isManuallySetCategory || overrideMap.has(overrideKeyForRow(row))) ? 1 : 0`.
Update the L2/reset doc-comments (they reference `row.hash`).

## Tests (adapt + 2 new probes)
- **Adapt** every 4.4 (`categorize-with-overrides.spec`, `auto-other.spec`?) + 4.7 (`auto-categorize.spec`)
  test that SEEDS an overrideMap with a bare `hash` key → seed with `overrideKey(hash, year, month)` (and
  give the rows a `date` whose UTC (year,month) matches). The 4.4 proofs (override survives re-import,
  L1>L2, isManual=0 re-evaluates, sandbox-independence, reset) + the 4.7 Q-007 contract + isManual emission
  must all PASS on the triplet key.
- **NEW probe — same-hash-different-month:** a row with the SAME hash as a seeded override but a `date` in a
  DIFFERENT (year,month) → the override does NOT apply (resolves via rules/null, `isManual=0`). Proves the
  narrowing; hash-alone would have mis-applied it.
- **NEW probe — derivation byte-match:** for a row, `overrideKeyForRow(row)` === `overrideKey(fp.hash,
  fp.year, fp.month)` where `fp = deriveFootprint(row, ...)` — a row and its footprint compute the SAME key.

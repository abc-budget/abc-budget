## Task 2 — `auto-other.ts` (completeness + dump fallback) + the TRANSIENT proof  [Wave A]
**Files:** `rules/auto-other.ts` (+spec).
- `effectiveCategory(row, ctx: OverrideContext, dumpCategoryId: string | null): Category | null` — SYNC:
  `resolveCategory(row, ctx) ?? (dumpCategoryId !== null ? ctx.categoriesById.get(dumpCategoryId) ?? null :
  null)`.
- `remainderRows(rows, ctx, dumpCategoryId): ImportStatementStage3Row[]` — rows where `effectiveCategory ===
  null`. (Before a dump, `dumpCategoryId = null` → remainder = `resolveCategory === null` rows.)
- `isComplete(rows, ctx, dumpCategoryId): boolean` — `remainderRows(...).length === 0` (the «Далі» gate).
- (NO `dumpRemainder` mutation — the dump IS just `dumpCategoryId` flowing into `effectiveCategory`; the
  caller sets its session `dumpCategoryId` and persists via Task 1. Optionally a tiny
  `confirmDump(categoryId, dao)` helper that `setLastRemainderCategoryId` + returns the id — keep it thin.)
- **Pins:**
  - «Далі» gate: a row set that resolves all-but-one → `isComplete === false`; give the last one a rule/dump
    → `true` (toggles on the LAST remainder row).
  - dump catches the remainder: `dumpCategoryId = X` → every previously-null row’s `effectiveCategory === X`;
    `isComplete === true`; remainder empty.
  - **rules beat the dump:** a row matched by a rule keeps the RULE category even with a `dumpCategoryId` set.
  - **L4 below rules / reclaim:** add a rule that newly matches a dumped row → its `effectiveCategory`
    becomes the rule category (dump reclaimed), without touching `dumpCategoryId`.
  - **TRANSIENT re-import proof (the headline):** `deriveFootprint(dumpedRow, usd,
    effectiveCategory(row,ctx,dumpId)?.id ?? null, 0)` → `footprintDao.put` (footprint `isManual:0`,
    `categoryId = X`); then a FRESH session (new `OverrideContext`, `dumpCategoryId = null`, no matching
    rule) → `effectiveCategory(reImportedRow) === null` (uncategorized again). The override map is UNTOUCHED
    (assert `isManual` stays 0 and the row is NOT in a re-loaded override map). Proves the dump is NOT sticky
    (vs a manual override).
  - dangling `dumpCategoryId` (id not in `categoriesById`) → `effectiveCategory` falls to `null` (fail-soft).

## Task 3 — `remainder-magnitude.ts` (decision B, fail-soft)  [Wave A]
**Files:** `rules/remainder-magnitude.ts` (+spec).
- `computeRemainderMagnitude(remainderRows, deps: { ratesService: ExchangeRateService; base: string }): Promise<{ opCount: number; baseTotal: number; uncachedTail: Map<string, number> }>` (opCount =
  remainderRows.length; the caller supplies totalOpCount for «N of M» — keep this fn focused on the sum).
  - Per row: if `row.currency === base` → `baseTotal += row.amount`. Else `await
    ratesService.convert(row.amount, row.currency, base, row.date)` (OPERATION date); on
    `RatesUnavailableError` → `uncachedTail.set(row.currency, (get ?? 0) + row.amount)` (FAIL-SOFT — never
    rejects the whole call). Any OTHER error → rethrow (only the uncached path is soft).
  - Construct the service in tests via `new ExchangeRateServiceImpl(new CacheOnlyRatesApi(dao))`.
- **Pins:** single-currency=base → exact `baseTotal`, empty tail; a cached cross-currency → converted into
  `baseTotal`; an UNcached currency → its amount in `uncachedTail` (NOT in `baseTotal`), and the call still
  RESOLVES (fail-soft); mixed → both; `opCount` correct; the rate read uses `row.date` (assert the convert
  call’s date arg / that seeding a DIFFERENT date leaves it uncached → tail).

## Confirmed inputs (PM rulings, binding)

1. **CONTRACT — NO bump.** `CONTRACT_VERSION` stays **3**. The worker self-derives the
   same-origin `/api/rates` endpoint internally; nothing about rates crosses the wire in
   `init`. No handshake field, no init param.
2. **PRIMARY KEY — compound keyPath `['hash','year','month']`.** Minimized identity *is*
   the key → native idempotent `put()` upsert. Exactly the 5 ENT-001 fields, no auto-id.
   The `hash` non-unique lookup index + the dedup write path are **3.4**, not here.
3. **Date asymmetry (design decision 1).** The store keeps `year`+`month` from the
   **operation** date (`row.date`), never the statement/export date. The hash keeps the
   full date with day (already true — hash is built in stage 3, unchanged here).
4. **USD is internal (ENT-020).** `amountUSD` is the reserve bridge only. No UI reads it.

## ⚠️ Direction correction folded into this plan (the MAJOR-2 / silent-money class)

The design spec's architecture sketch wrote `amountUSD = amount × rate`. **That is the
wrong direction.** Mapped from the live code:

- The OER proxy returns **USD-base** rates. They are stored `base='USD'` and the rate map
  is **units-of-currency per USD** — e.g. `rates['UAH'] = 41` means *1 USD = 41 UAH*.
- The already-tested `ExchangeRateServiceImpl.convert()`
  ([`service.ts:88-89`](../../../packages/engine/src/internal/exchange-rate/service.ts))
  proves the direction: USD→other is `amount * rates[other]`; **other→USD is
  `amount / rates[from]`** (division).
- So 100 UAH at rate 41 → **100 / 41 ≈ 2.44 USD**. A multiply (`× 41`) would yield
  4100 USD — a silent 1680× money error that is fully deterministic and would pass any
  self-consistency test.

**Design consequence:** `toAmountUSD` does **not** hand-roll the multiply. It delegates to
the existing, tested `service.convert(amount, currency, 'USD', date)` — the single source
of truth for direction — backed by a **cache-only** rates API so there is no network at
commit time. The known-value direction test (Task 6) pins this against a concrete number.

- v1 (no-op anchor), v2 (`exchangeRates`), v3 (`userSettings` + `recallPool`) steps are

  - The three prior stores still exist with unchanged keyPaths (no regression).

### Task 2 — `FootprintRecord` type (exactly 5 fields, minimization pinned at type level)

**Files:** `footprint/types.ts`; `footprint/types.spec.ts` (type-level + a runtime
key-count guard).

```ts
export interface FootprintRecord {
  readonly year: number;        // operation-date year   (UTC)
  readonly month: number;       // operation-date month  1–12
  readonly amountUSD: number;   // ENT-020 reserve bridge — internal only
  readonly categoryId: string | null;  // null until EP-4 populates it
  readonly hash: string;        // dup-wrapped final hash from stage 3 (row.hash)
}
```

- **Pins:** a compile-time exact-key check (e.g. a `KeysEqual<FootprintRecord, 'year'|'month'|'amountUSD'|'categoryId'|'hash'>` assertion) so adding a 6th field fails the
  build (HC-2/3, VIS-002). A short doc comment states the minimization contract and that
  raw identifying text lives only inside `hash`, never readable in storage.

### Task 3 — `deriveFootprint(row, amountUSD)` (pure, operation-date split)

**Files:** `footprint/derive-footprint.ts`; `footprint/derive-footprint.spec.ts`.

- Pure function: `deriveFootprint(row, amountUSD): FootprintRecord`.
  - `year` / `month` from **`row.date` (the operation date)** — UTC, `month` 1–12.
  - `categoryId = null` (EP-4 populates later).
  - `hash = row.hash` (already the 3.2 dup-wrapped final hash; do not re-hash).
  - `amountUSD` passed in (computed by Task 6; kept separate so derive stays
    rate-free and synchronous).
- **Pins:** decision-1 asymmetry — a row whose operation date and statement/export date
  differ in month must yield the **operation** month. Year/month boundary cases
  (Jan, Dec, single-digit month zero-handling). Output has exactly the 5 keys.

### Task 4 — `FootprintDao` (basic put + getAll; NOT the dedup batch path)

**Files:** `footprint/footprint-dao.ts`; `footprint/footprint-dao.spec.ts` (fake-indexeddb).

- An `IDBBatchDao`-style DAO over the `footprint` store: `put(record)` and `getAll()`.
- **Native idempotency pin (the keyPath proof, set up for 3.4):** two `put()`s of records
  sharing the same `['hash','year','month']` leave `getAll().length === 1` and the second
  value wins (overwrite). Two records differing in any key element → 2 rows.
- **Scope fence:** the multi-record upsert/dedup *batch* write path, the zero-growth
  re-import proof, the count-matches-bank proof, and the `hash` lookup index are **3.4** —
  this task stops at single-record `put` + `getAll`. State that in the file header.

### Task 5 — `WorkerHttpRatesApi` + `warmRates(dates)` (self-origin, no contract touch)

**Files:** `exchange-rate/worker-http-rates-api.ts`;
`exchange-rate/worker-http-rates-api.spec.ts`; composition-root wiring (Task 7).

- `WorkerHttpRatesApi implements ExchangeRateApi`: same request shape as the app's
  `HttpRatesApi` — `fetch(import.meta.env.VITE_RATES_URL ?? '/api/rates', { POST, {date} })`
  — but worker-side (same-origin, Vite inlines env). Fail-loud on non-OK / network error.
- `warmRates(dates: Date[])`: pre-fetches and caches the USD rate rows for the distinct
  operation dates of a batch via the cached read-through, so that commit-time
  `toAmountUSD` reads hit the cache. **This is the only network touch, and it is OFF the
  import path** (import stays rate-free, EP-2 proven offline).
- **Pins:** request URL + body shape match `HttpRatesApi` (parity test); non-OK response
  throws (fail-loud, HC-7); `warmRates` is idempotent against the cache (warming an
  already-cached date does not re-fetch). No `CONTRACT_VERSION` reference anywhere.

### Task 6 — `toAmountUSD(...)` — cache-only, fail-loud, **known-value direction test**

**Files:** `footprint/footprint-usd.ts`; `footprint/footprint-usd.spec.ts`;
a small `exchange-rate/cache-only-rates-api.ts` (reads DAO, throws on miss — never
delegates to remote).

- `toAmountUSD(amount, currency, opDate, cacheOnlyService): Promise<number>` delegates to
  `cacheOnlyService.convert(amount, currency, 'USD', opDate)`. **Direction is owned by the
  tested `convert()`** — we do not re-implement the multiply/divide.
- The cache-only path: `CacheOnlyRatesApi` reads `dao.findByBaseCurrencyAndDate('USD',
  fmtDate)`; on a **cache miss it throws `RatesUnavailableError`** (HC-7 fail-loud — never
  returns 0, never a nearest-date guess, never a network fetch). `toAmountUSD` propagates
  it. There is **no `fetch` reachable from this function** — assert that (offline-safety
  pin).
- **THE KNOWN-VALUE DIRECTION TEST (PM-required, test-with-teeth):**
  - Seed the cache with a realistic **USD-base** row: `base='USD'`, a date, `UAH = 41.0`
    (i.e. 1 USD = 41 UAH — units-per-USD, the OER convention).
  - Assert `toAmountUSD(100, 'UAH', date) ≈ 2.4390` (`toBeCloseTo`, 100 / 41).
  - **Teeth:** assert the result is **NOT** `4100` (the wrong-direction multiply) and
    **NOT** `100`. A reversed multiply/divide, or a no-op, fails the test. The test
    comment states the convention explicitly: *stored rate is units-of-target-currency per
    USD; other→USD divides.*
  - Add a second realistic pair (e.g. `EUR = 0.92`, 100 EUR → ≈ 108.70 USD) so the test
    pins direction across a rate both >1 and <1 — a single multiply-vs-divide swap cannot
    pass both.
  - **Determinism pin:** same inputs → byte-identical output across repeated calls
    (HC-9). Determinism is asserted *in addition to* the known value — not instead of it.
  - **Fail-loud pin:** uncached date → rejects with `RatesUnavailableError`; the rejection
    message names the missing date.

### Task 7 — Composition-root wiring (self-derive, no init param)

**Files:** `internal/worker/composition-root.ts` (+ its spec); confirm
`apps/web/src/engine.ts` needs **no** change to the wire contract.

- In `composeEngine`, wire the rates remote via `setRemoteRatesApi(new WorkerHttpRatesApi())`
  — **self-derived, same-origin.** This closes the 2.6 carry-forward gap (worker had a
  cache DAO but no remote) **without** an `init` param and **without** touching
  `CONTRACT_VERSION`.
- **Pins:** `composeEngine()` called with no rates option still wires a working remote
  (the self-derived one); `CONTRACT_VERSION` is still `3` (assert the literal is
  unchanged); the worker handshake schema is untouched.

## Cross-cutting Definition of Done

- [ ] Migration v4 creates `footprint` with keyPath `['hash','year','month']`; v1–v3
      byte-unchanged; **fresh ≡ upgrade** re-proven; prior stores intact.
- [ ] `FootprintRecord` is exactly 5 fields, enforced at compile time (6th field → build
      fails). IndexedDB inspection of a footprint row cannot reconstruct an operation
      (no raw description/counterparty/amount/currency — only year, month, amountUSD,
      categoryId, opaque hash).
- [ ] `deriveFootprint` takes year/month from the **operation** date (decision-1 asymmetry
      pinned by a differing-statement-date test).
- [ ] `toAmountUSD` gets the **direction right** against a concrete known value (100 UAH @
      41 → ≈ 2.44 USD, **not** 4100); fails on a reversed multiply/divide; is deterministic;
      and **does no network I/O** (cache-only, fail-loud `RatesUnavailableError` on miss).
- [ ] `warmRates` is the only network touch, **off the import path**; import stays offline
      (no EP-2 regression).
- [ ] `WorkerHttpRatesApi` matches the app `HttpRatesApi` request shape and fails loud.
- [ ] `CONTRACT_VERSION` is still **3**; no `init` rates param; handshake untouched.
- [ ] `FootprintDao` does single-record `put` + `getAll` with the native-idempotency pin;
      the dedup **batch** write path, re-import proofs, and `hash` index are explicitly
      deferred to **3.4** (stated in file headers, not silently omitted).
- [ ] Full engine test suite green; typecheck + lint clean.

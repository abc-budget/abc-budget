## Task 1 — Migration **v5**: `hash` non-unique index on `footprint`

**Files:** `persistence/engine-db.ts` (append v5 step + update the lineage comment);
`persistence/persistence.spec.ts` (extend the migration proofs).

- Append ONE step `{ toVersion: 5 }` calling
  `ctx.createIndex('footprint', { name: 'hash', keyPath: 'hash', options: { unique: false } })`.
  The store already exists (v4) — `createIndex` on the upgrade tx is the supported path (no `rebuildStore`).
- v1–v4 steps **byte-unchanged**. Update the comment block to document v5.
- **Pins:** after open at latest version, the `footprint` store has an index named `hash`,
  `keyPath === 'hash'`, `unique === false`; the store keyPath is still `['hash','year','month']`;
  **fresh ≡ upgrade** (fresh v1→v5 and an upgrade from a v4 DB both end with the `hash` index) —
  reuse the existing invariant harness; prior stores (`exchangeRates`/`userSettings`/`recallPool`)
  intact; the step-count anchor updated 4→5.

## Task 2 — `FootprintDao`: `findByHash` + single-transaction `putBatch`

**Files:** `footprint/footprint-dao.ts` (+ `footprint-dao.spec.ts`). Update the file-header SCOPE
FENCE (3.4 now implements these; the index is created by migration v5).

- `findByHash(hash: string): Promise<FootprintRecord[]>` → delegates to the base
  `findByIndex('hash', hash)`. (Each op's final dup-wrapped hash maps to exactly one row, but the
  return stays an array — the index is declared non-unique.)
- `putBatch(records: readonly FootprintRecord[]): Promise<void>` → open **one** `readwrite`
  transaction on `footprint`, `store.put(r)` every record, resolve on `transaction.oncomplete`,
  reject on `transaction.onerror`/`onabort`. Atomic: a mid-batch failure aborts the whole tx
  (zero writes). Empty input resolves without opening a write.
- **Pins:** `findByHash` returns the row for a present hash, `[]` for an absent one; `putBatch`
  writes N rows in one tx (`getAll().length === N`); **idempotency** — `putBatch` of the same
  triples again leaves the count UNCHANGED, last value wins; `putBatch([])` is a no-op.

## Task 3 — `commitFootprints` write path (two-phase, fail-loud)

**Files:** `footprint/commit-footprints.ts` (+ `commit-footprints.spec.ts`).

- `commitFootprints(rows: readonly TransactionRow[], deps): Promise<CommitResult>` where
  `deps = { footprintDao: FootprintDao, ratesDao: ExchangeRateDAO, warm: (dates: Date[]) => Promise<void> }`.
  (`warm` injected so tests/proofs control the network; production wiring in EP-5 passes
  `WorkerHttpRatesApi().warmRates`. Default may be a no-op or the real warm — keep it injectable.)
- **Orchestration (two-phase, all-or-nothing):**
  1. **warm:** `await warm(distinctOpDates(rows))` — best-effort; swallow a network/HTTP failure
     (the cache-only convert below is the single canonical loud gate). The ONLY network touch.
  2. **pre-flight (pure, NO writes):** for each row, `amountUSD = await toAmountUSD(row.amount,
     row.currency, row.date, ratesDao)` (cache-only) → `deriveFootprint(row, amountUSD)`. A single
     uncached rate throws **`RatesUnavailableError`** → propagate → **whole commit aborts, ZERO writes**.
  3. **write:** `await footprintDao.putBatch(records)` — one transaction, native idempotent upsert.
- Return a small `CommitResult` (e.g. `{ written: number }`) — internal, no wire shape.
- **Pins:** happy path writes every row (`getAll().length === rows.length` after distinct keys);
  **one uncached date → rejects `RatesUnavailableError` AND `getAll()` is empty (zero writes)** —
  the headline two-phase proof; re-commit of the same rows leaves the count unchanged (idempotent);
  deterministic; no `fetch` reachable when the cache is warm (convert is cache-only).

## Task 4 — THE PROOFS (the EP-3 payoff — real tests)

**Files:** `footprint/dedup-proofs.spec.ts` (synthetic, always-run);
`footprint/real-statement-proof.spec.ts` (gated-skip).

Drive **real generated rows** through `generateRows` (so hashes come from the genuine 3.2 recipe —
`applyDupCounters` + `pseudoOp`), seed the rate cache for the fixture dates (or stub `fetch` per the
3.3 seam harness), `commitFootprints`, assert store counts. Use the `createMockRow` / `ColumnInfo`
helper pattern from `stage3/row-generator.spec.ts`; the pseudo-op trio mirrors its
`BANK_COMMISSION`+`CASHBACK` mapping (`row-generator.spec.ts:486`).

**Synthetic proofs (always-run, CI):**
- **zero-growth re-import:** generate a statement's rows, commit, then commit the SAME again →
  `getAll().length` UNCHANGED.
- **overlap (union, not sum):** commit a Jan–Mar set, then a Jan–Dec set that re-includes the Jan–Mar
  ops → final count = the UNION (overlapping ops counted once, native upsert).
- **in-month true-duplicate (FEAT-008):** two byte-identical ops in one month → **TWO** records
  (3.2 dup-counter → distinct hashes → distinct keys → both kept, never merged; count reconciles
  with the bank).
- **pseudo-op trio:** one source row with commission + cashback → **THREE** records (main +
  commission + cashback; distinct `pseudoOp` → distinct hashes → distinct keys).

**Real statement (gated-skip, local-only):**
- Read a real **mono UA** statement from outside the repo — path via `process.env.ABC_REAL_STATEMENT`,
  falling back to the known local `D:\abc-budget\mono_07-10-23_14-34-50.csv`. **`it.skip`/early-return
  when the file is absent** (CI has no such file) — `log` what was skipped (HC-10, no silent skip).
- Full pipeline (decode → stage2 mapping → `generateRows`) → `commitFootprints` **twice** → assert
  **zero growth** on the second commit; surface the count of any genuine in-month duplicates found
  (informational — real data legitimately contains some).
- NEVER commit the statement or its contents (real financial data; OSS repo).

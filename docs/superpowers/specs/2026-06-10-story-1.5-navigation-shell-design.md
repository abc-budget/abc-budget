## Goal

The routing + shell skeleton per FEAT-030: `/` routes by a stubbed `hasData`; two dwell zones
(Dashboard·Settings) with an identical header zone-switcher; Import mounted as a **launched flow**
(own stepper, no zone-switcher); Settings section-tabs as an in-page view switch; logo→Dashboard
everywhere except inert on Onboarding; **no dead-ends**; **no review state-switcher**. Screens are
placeholders — later epics fill them; the navigation skeleton + link map land now.

## Decisions (resolved with the founder)

1. **Router: react-router v7, library mode** (BrowserRouter + Routes). The route surface grows
   exactly where minimal routers start costing (EP-2 wizard gates, FEAT-030 deep-links, guards);
   ~12KB gzip is negligible vs the font precache; boring standard for an OSS Core.
2. **Wizard steps: single `/import` route, internal step state.** Gates = state transitions —
   URL bypass structurally impossible (HC-8 safe by construction); the stepper IS the navigation
   per FEAT-030; explicit «Назад» keys step back. Browser Back exits the flow (consistent with
   "wizard always exits to Dashboard"). *Carry-forward → EP-2:* mid-wizard exit protection
   (`useBlocker` confirm-on-leave) once real parsed work lives in the flow; step-URLs revisited
   only if a real need appears.
3. **Nav chrome: presentational in `altus/`, wiring in the app layer.** ZoneSwitcher/SectionTabs/
   Stepper join `altus/components` as pure presentational components (items/active/renderItem or
   callback props — no router imports; the 1.3 ESLint fence stays intact); their CSS joins
   `altus-components.css` under the same tokens-only extraction discipline. `src/app/` owns
   headers, routes, and the hasData stub. All three chrome components join `/gallery.html`.

## Prototype facts (verified)

- Zone-switcher: `.zone-nav` (dark inset `#0E2630` plate) / `.zone` / `.zone.on` — identical CSS in
  `Dashboard.html` and `Settings.html`.
- Section tabs: `.subnav-tab` / `.subnav-tab.on` — light, gold-underline, visually distinct from
  the dark zone-switcher (per FEAT-030).
- Stepper: `.stepper` / `.stp` (`.on`/`.done`) / `.stp-dot` / `.stp-lab` / `.stp-rule` in
  `Import-Source.html`; mobile collapses the stepper and shows the `.ob-step-m` «КРОК N / 4»
  eyebrow (media query — CSS-only behavior).

## 2. Routes & the redirect

- `/` = `useHasData() ? <Navigate to="/dashboard" replace> : <Onboarding/>` — Onboarding lives
  **at root only** (first-run home *is* root; no separate URL).
- `/dashboard`, `/settings`, `/import` — plain routes. Unknown paths → `<Navigate to="/" replace>`
  (no-dead-ends includes typos).
- `has-data.ts`: `useHasData(): boolean` returning a module constant `false`, with the EP-3 seam
  documented (the real predicate asks the engine once footprints exist). Router tests mock the
  module both ways.
- Logo: `BrandMark href="/dashboard"` on all product screens (rendered as a router `<Link>` via
  the header, not a full-page `<a>` reload — see §3); inert `BrandMark` on Onboarding (FEAT-030).

## 3. Headers (the thing 1.4 is waiting for)

Three compositions in `app/headers.tsx`, **all** carrying a language-toggle slot (an empty
`<div data-slot="lang" />` placeholder immediately right of the zone-switcher/stepper, per
Navigation.md "on every screen" — 1.4 replaces the placeholder with the real toggle):

- **DwellHeader** (Dashboard, Settings): linked brand · `ZoneSwitcher` `[Дашборд | Налаштування]`
  (active zone highlighted; byte-identical composition on both dwell screens) · lang slot.
- **FlowHeader** (Import): linked brand · `Stepper` (ФАЙЛ · КОЛОНКИ · КАТЕГОРІЇ · ОГЛЯД; CSS
  collapses to the «КРОК N / 4» eyebrow on mobile) · lang slot. **No zone-switcher.**
- **OnboardingHeader**: inert brand · lang slot. No zone-switcher.

Brand-as-Link note: `BrandMark` (1.3) renders a plain `<a>` when given `href`. For SPA navigation
the headers wrap an inert `BrandMark` in a router `<Link className="brand-link">` (or pass
`renderItem`-style composition) — implementation detail; the visible markup keeps `.brand` styling.

## 4. Chrome component APIs (altus/, fence-compliant)

```ts
ZoneSwitcher({ items: Array<{ id: string; label: string }>, activeId: string,
               renderItem: (item, active) => ReactNode })   // .zone-nav wrapper; app injects <Link class="zone [on]">
SectionTabs({ tabs: Array<{ id: string; label: string }>, activeId: string,
              onSelect: (id: string) => void })             // .subnav-tab buttons; pure callback
Stepper({ steps: Array<{ id: string; label: string }>, activeIndex: number,
          stepWord?: string })                              // .stp on/done by index; mobile «КРОК N / 4» eyebrow
```

No router imports anywhere in `altus/` — the existing ESLint fence enforces it. CSS extracted
faithfully from the prototypes into `altus-components.css` (tokens only; the recurring `#0E2630`
plate literal stays, documented same as the gauge/CRT).

## 5. Wizard flow shell (single `/import` route)

`ImportFlow.tsx`: `stepIndex` state behind the Stepper; four placeholder step bodies (S3a Джерело ·
S3b Колонки · S3c Категоризація · S3d Огляд — each a `Panel` naming its EP-2/EP-3 story). Footer
keys per the FEAT-030 map:

- «Далі / Продовжити» → next step (gate stubs are **always-open but stateful** — the transition
  function exists and EP-2 swaps in real gate predicates; no URL can bypass them by construction).
- «Назад» → previous step, **except S3a where «Назад» → `/dashboard`**.
- S3d: «До бюджету» → `/dashboard`; «Імпортувати ще» → reset to S3a (state reset, stays on `/import`).

## 6. Link map & no-dead-ends (the acceptance)

All Navigation.md map rows wired: Onboarding «Імпортувати виписку» + «Спробувати на прикладі» →
`/import` · Dashboard «Імпорт виписки» CTA → `/import` · Settings DAT-panel «Імпорт виписки» →
`/import` · zone-switcher Dashboard ↔ Settings · logo per §2 · wizard exits per §5 · Settings tabs
in-page.

**Router tests (MemoryRouter) assert each row,** plus the invariants:
- `/` redirect both ways (mocked `hasData` false → Onboarding content; true → Dashboard).
- Zone-switcher present on exactly the two dwell screens; absent on Onboarding and Import.
- Logo inert exactly on Onboarding; links to `/dashboard` elsewhere.
- Wizard: Далі walks all four steps; Назад steps back; S3a-Назад exits to Dashboard; both S3d
  exits behave; «Імпортувати ще» resets to step 0.
- **No-dead-ends invariant:** every screen renders ≥1 outgoing navigation affordance.
- **No review state-switcher** rendered anywhere (regression grep + render assertion).

Chrome components get unit render-tests (zone `.on` class, renderItem injection, tab onSelect,
stepper on/done classes + eyebrow text). The 1.1 ping/version slice moves onto the Dashboard
placeholder as a CRT status line («ENGINE 0.0.0 · CONTRACT 1 · PING OK») — the boundary proof
stays visible and its test moves with it.

## 7. Out of scope / carry-forward

- Real screen content: Dashboard (EP-6/7), Settings panels (EP-5/6), wizard step bodies (EP-2/3).
- The language toggle itself → **1.4** (fills the `data-slot="lang"` placeholder in all three headers).
- Wizard gates (real predicates) + `useBlocker` exit-protection → EP-2.
- **Settings deep-link (Dashboard category card → Settings → Категорії) → EP-6: prefer a query
  param (`/settings?tab=categories`) over route state so the deep-link is bookmarkable** *(founder

- Base-currency gate in S3a → 1.6/EP-2. RUL//LOG/ sub-surfaces → EP-2 (inside S3c).
- Review state-switcher → never (production).

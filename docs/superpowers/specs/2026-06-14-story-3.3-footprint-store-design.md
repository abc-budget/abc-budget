## Goal

The first durable, minimized operation store: a new `footprint` IDB store (new append-only migration step on the 1.2 framework) holding ENT-001's EXACT 5 fields `{ year, month, amountUSD, categoryId, hash }` and **nothing else**. Plus the USD conversion that derives a footprint from a typed `TransactionRow` — `amountUSD = amount × OER-rate(operation date)` — **without regressing the EP-2 offline import** and **without a silently-wrong/zero USD**. REBUILD (the prior-art store persists full operations; ABC's is minimized by design).

## Locked decisions (PM, 2026-06-14) — carried

1. **Date asymmetry, enforced (ENT-001/ENT-014):** the store keeps ONLY `year+month`; the hash (3.2) keeps the full date with day. `year/month` come from the **OPERATION date**, NOT the statement/export date. PIN: a fixture where statement-date ≠ operation-date proves the month is taken from the operation.
2. **USD needs rates reachable IN THE WORKER** (the remote-rates carry-in); conversion deterministic (HC-9); rate unavailable → fail loud (HC-7), never zero/silent. **(Mechanism re-resolved below — see §1.)**
3. **footprint-USD is the reserve bridge only (ENT-020)** — not a display value; no UI reads it.
4. **Contract-surface awareness:** if it touches the EngineClient wire/handshake, it's a declared `CONTRACT_VERSION` v3→v4 bump (exact-match) — must be flagged, not slipped. **(Resolved below: NO bump needed — see §1.C.)**

## 1. THE FORK — rate sourcing, offline-safety, contract (resolved with evidence)

**The latent conflict the PM flagged:** decision 2 as literally written ("the client passes the `/api/rates` base in init; the worker fetches at write") implies (a) a wire change → contract bump, and (b) a network fetch on the import/commit critical path → the EP-2 offline-import regresses. Both are avoidable. Resolution below, grounded in the existing rate infra.

### Evidence — the existing rate infra (mapped)

| Piece | Fact (file) |
|---|---|
| `exchangeRates` IDB store | date-keyed: keyPath `['base','date']` (+ `base`,`date` indexes), date = `"yyyy-MM-dd"` (`exchange-rate/dao.ts`). A direct `findByBaseCurrencyAndDate(base,date)` READ is offline-safe (no fetch). |
| `CachedExchangeRateApi` | read-through: checks the DAO cache; on miss delegates to the remote + caches (`cached-api.ts`). |
| `getRatesService()` | returns **null** when no remote api is wired (`rates-holder.ts`). The worker has NO remote api today (the 2.6 carry-forward) → it can READ the cache but cannot fetch. |
| `HttpRatesApi` (app) | just `fetch('/api/rates', {date})` POST → same-origin Hosting rewrite → the 2.2b OER-proxy Cloud Function. `RATES_URL = import.meta.env.VITE_RATES_URL ?? '/api/rates'` (`apps/web/src/app/rates/http-rates-api.ts`). |
| The worker | same-origin (Workers must be); Vite bundles `@abc-budget/engine/worker` as a worker chunk **with the app's env inlined** (`engine-worker.ts`). So a worker-side `fetch('/api/rates', …)` resolves against the app origin, and `import.meta.env.VITE_RATES_URL` is available in the worker bundle. |
| EP-2 import path | decode → map → `generateRows` produces typed rows in the **original** currency — **rate-free**. The EP-2 offline proof (10.5k rows offline) covers this path; it never touched rates. The footprint conversion is the FIRST rate consumer, and it lives AFTER generate (at commit). |

### A. Rate sourcing — read the cache; warm separately; never fetch on the write

- **The footprint WRITE READS the cached rate** for the operation's date directly from the `exchangeRates` DAO — offline-safe, deterministic (`amountUSD = amount × rate`, pure). It NEVER fetches.
- **A separate rate-WARM step** fetches+caches the rates for the batch's operation dates — the ONLY network touch, and it is NOT on the EP-2 decode/map/generate path. It runs when online (at commit-prep / when the operation dates are known).
- **Missing rate at write → fail loud, block (HC-7):** if the cache has no rate for an operation's date, the conversion does NOT write a zero/guessed USD — it raises a structured `RatesUnavailableError` naming the missing dates. The commit is blocked with "connect once to fetch rates for [dates]" (the UI copy is EP-5; 3.3 surfaces the structured error).

**Recommended option (of the PM's four):** a composite — **rate-warm (online) populates the cache + read-cache-at-write + fail-loud-block on uncached.** Rejected: *defer-USD-and-backfill* (a null-USD footprint is an incomplete reserve bridge + a non-minimized transient + backfill complexity); *nearest-cached-rate* (wrong rate — violates correctness/determinism intent). This satisfies all three constraints: **(1)** the EP-2 offline IMPORT never regresses (it's rate-free; only the new commit-time conversion needs rates, and offline+uncached fails loud — which is correct: you cannot compute USD without ever having fetched a rate); **(2)** never silently-wrong/zero; **(3)** deterministic.

### B. Wiring rates into the worker — the worker self-derives, no wire param

The 2.6 gap ("the app-side HTTP rates api cannot cross the wire as a function") is resolved NOT by serializing the function, but by the worker **reconstructing the same fetch logic internally**: a worker-side `WorkerHttpRatesApi` doing `fetch(import.meta.env.VITE_RATES_URL ?? '/api/rates', {date})` — same code, same same-origin endpoint, same 2.2b proxy. The composition root wires it into `setRemoteRatesApi(...)` so the cache CAN be warmed. The proxy origin allowlist is satisfied (a worker fetch sends `Origin: <app origin>` = the allowlisted prod host; dev uses the `FUNCTIONS_EMULATOR` relaxation). In node/tests (no origin/proxy) the worker composes with NO remote (cache-only) — exactly today's behavior.

### C. Contract — NO `CONTRACT_VERSION` bump

Because the worker self-derives the same-origin `/api/rates` (no init param, no `hello`/handshake field, no new method), **the EngineClient wire is unchanged → `CONTRACT_VERSION` stays 3.** I recommend AGAINST decision 2's literal "pass the base in init" (it forces an avoidable v3→v4 bump for zero benefit — the worker already knows its origin, and the `VITE_RATES_URL` override is inlined at app build). **Flagged alternative:** IF the PM wants the rates base passed explicitly via init (e.g. for a reason beyond what's mapped here), THAT is a declared v3→v4 exact-match-handshake bump — call it out; I do not recommend it.

## 2. Secondary fork — the footprint store's primary key (recommend, flag)

ENT-001 is exactly `{ year, month, amountUSD, categoryId, hash }` — no `id` field. The store needs a primary key derived from these. Two shapes:

- **(Recommended) compound keyPath `['hash','year','month']`** — the minimized identity IS the key; an IDB `put()` on an existing key OVERWRITES → idempotent dedup natively, no extra field, no separate index. 3.4 then builds the upsert WRITE PATH (batch commit) + the dedup proofs (zero-growth re-import, count-matches-bank) on top of this native idempotency.
- (Alternative) auto-increment `id` + a separate `(hash,year,month)` **unique index** — adds a non-ENT-001 `id` field to the store and a conflict-then-update path. More machinery; introduces a field beyond the 5 (minimization tension).

I recommend the compound keyPath (stays exactly the 5 fields; native upsert). 3.4's assigned "unique index + upsert dedup" is then realized as the keyPath's uniqueness + the commit/dedup write path. **Flag for PM:** confirm the compound-keyPath choice (vs auto-id + unique index) — it shapes 3.3's migration and 3.4's scope.

## 3. Architecture (3.3 builds; 3.4 = upsert/dedup proofs; EP-5 = the commit UI)

```
packages/engine/src/internal/
├─ persistence/engine-db.ts        # MIGRATION v4 (append-only): createStore('footprint',
│                                  #   { keyPath: ['hash','year','month'] }). v1/v2/v3 byte-unchanged;
│                                  #   fresh≡upgrade re-proven at v4 (the 1.2 invariant test).
├─ footprint/
│  ├─ types.ts                     # FootprintRecord = { year, month, amountUSD, categoryId, hash }
│  │                               #   — EXACTLY 5 fields (HC-2/3; a type-level pin that nothing else leaks)
│  ├─ footprint-dao.ts             # IDBBatchDao on the footprint store (the 1.2/2.3 pattern); basic put +
│  │                               #   getAll (the upsert/dedup batch path is 3.4)
│  ├─ derive-footprint.ts          # deriveFootprint(row, amountUSD): TransactionRow → FootprintRecord.
│  │                               #   year/month from the OPERATION date (row.date) — decision 1.
│  │                               #   categoryId = null (EP-4). hash from row.hash (already dup-wrapped, 3.2).
│  └─ footprint-usd.ts             # toAmountUSD(row, ratesReader): reads the CACHED rate for row.date;
│                                  #   amountUSD = amount × rate; MISSING → RatesUnavailableError (HC-7).
│                                  #   currency already ISO; rate base/target per the 1.6 model.
└─ exchange-rate/
   └─ worker-http-rates-api.ts     # NEW: WorkerHttpRatesApi (same fetch as HttpRatesApi, worker-side) +
                                   #   a warmRates(dates) that fetches+caches uncached dates (online only).
                                   #   Composition root wires it via setRemoteRatesApi (self-origin).
```

- **No contract/DTO change** — the footprint write is engine-internal (commit lands in EP-5; 3.3 provides the derivation + conversion + store + DAO + the worker rates wiring). `CONTRACT_VERSION` frozen at 3.
- **Minimization is enforced at the TYPE and the WRITE:** `FootprintRecord` has exactly 5 fields; `deriveFootprint` constructs ONLY those; a test pins that no raw field (description/counterparty/mcc/account/bankCategory/exact-amount/currency/day) appears in a stored record (Application→IndexedDB inspection cannot reconstruct an op).

## 4. Tests / pins

- **Migration v4:** v1/v2/v3 byte-unchanged; v4 creates `footprint` with the keyPath; fresh-install ≡ upgrade (the 1.2 invariant at v4); the existing 3 stores untouched.
- **Minimization (HC-2/3 PIN):** a derived+stored record has EXACTLY the 5 keys; no raw identifying field present; type-level + runtime assertion.
- **Decision-1 PIN:** a fixture where statement/export date ≠ operation date → `year/month` taken from the OPERATION date (`row.date`).
- **USD conversion:** cached rate present → `amountUSD = amount × rate` (deterministic, HC-9, same op+rate→same USD); rate MISSING → `RatesUnavailableError` naming the date; NO write, NO zero/silent (HC-7).
- **Offline-safety PIN:** `toAmountUSD` performs NO network fetch (reads the cache only) — the EP-2 import path stays rate-free; assert the conversion uses the DAO read, not a fetch.
- **Worker rates wiring:** the worker self-constructs the rates api from the same-origin path; `warmRates` fetches+caches uncached dates; cache-only read works with no remote (node/test).
- Determinism greps (no clock/random in the new code beyond classified sites); cold gate.

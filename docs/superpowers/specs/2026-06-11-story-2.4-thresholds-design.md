# Story 2.4 — Column-error thresholds + rollback — Design

> **Epic:** EP-2 · **Story:** 2.4 · **Date:** 2026-06-11 · **Branch:** `story/2.4-thresholds` (off `main`, post #11)
> **Source prompts:** `requirements/deliverables/2026-06-10-EP-2-ingestion-dev-prompts.md` (Story 2.4)
> **KB:** ENT-015 (>30% transform errors → type NOT applied, rollback + loud "why"), ENT-016
> (engine params 90 / 0.3 / 0.8; NFR-009: never raw in UI), Q-009 (unmapped column STOPS the
> flow explicitly), NFR-009 (low-level params behind safe abstractions), FEAT-022 (complete
> errors, not first), VIS-008 (transparency).
> **Flow:** execution → dev final review → STOP, hand back to founder (role fence).

## Goal

The ENT-016 thresholds move from 2.2's static constants into the settings store 2.3 landed
(sync read surface preserved via a hydrated snapshot, frozen per import session); the >30%
rejection carries its evidence (typed exception with ALL per-cell errors — ЩО/ЧОМУ/ДІЯ); the
Q-009 stop becomes explicit and structured (typed error + getter off one predicate).

## Current state (what 2.2/2.3 already laid)

- `validateErrorPercentage` (column.ts) already gates at >30% and throws BEFORE the column
  commits — rollback-to-UNKNOWN is already structural (immutable copy; on throw the original
  unmapped column is never replaced). The gap: the per-cell errors collected during the failed
  transform are DISCARDED on throw — the exception carries only the aggregate %.
- `stage2.next()` already stops on unmapped columns — but with a generic non-localizable Error
  that doesn't enumerate which columns.
- `getEngineConfig()` has 6 call sites, ALL synchronous (incl. recall's sync status heuristic);
  the 2.3 settings store is async IDB. No new migration needed — `userSettings` (v3) holds the
  override rows; ENGINE_MIGRATIONS untouched.

## Locked decisions (founder, 2026-06-11)

1. **Config architecture: hydrated snapshot.** `getEngineConfig()` stays sync over a
   module-level snapshot = ENT-016 defaults ⊕ store overrides. `hydrateEngineConfig(dao)`
   loads overrides; `setEngineParam(dao, key, value)` validates (loud throw on out-of-range —
   the NFR-009 teeth) and writes the STORE ONLY. **Session consistency refinement: the
   snapshot mutates ONLY at hydrate.** Hydrate runs at engine init and at import-session
   start; a `setEngineParam` mid-import applies from the NEXT session, never mid-flow.
   Pinned test: write during an active session → current session's transforms keep the old
   value; next session sees the new one. Multi-tab stale window: accepted (single-user
   local app — on record). All 6 call sites untouched.
2. **Rejection payload: structured exception.** `ColumnTransformRejection extends
   LocalizableException` carrying `{ errorCount, totalCount, threshold, cellErrors:
   [{rowIndex, error}] — ALL of them (FEAT-022 complete-not-first) }`. ЩО = class + counts ·
   ЧОМУ = the complete per-cell list · ДІЯ = localizable action hint in the message. The
   ported throw contract stays INTACT (1,347-line suite unweakened; the gate site's rebuild
   is a declared, hunk-audited adaptation). **Boundary pin: this throw is a per-COLUMN loud
   event, NOT a file-level halt.** Pinned test: one >30%-bad column rejected → that column is
   UNKNOWN + payload complete; sibling columns' transforms unaffected; session alive.
3. **Q-009 stop: typed error + getter, ONE predicate.** `next()` throws
   `UnmappedColumnsError` (LocalizableException subclass) enumerating unmapped column
   names/ids; stage2 exposes pure `getUnmappedColumns()` for 2.8's Option-A gate to render
   without provoking the throw. Both derive from the SAME predicate — pinned test:
   `getUnmappedColumns().length === 0 ⟺ next() does not throw`, and the error's enumeration
   equals the getter's list. Engine-internal until 2.6.
4. **Error-% denominator: ALL rows, faithful to prior art.** Ported suite's pinned numbers
   stand; the 2.1 decoder already strips trailing summary/empty rows pre-stage2. KB WATCH:
   a statement whose BODY is dense with empty rows makes the gate more permissive — revisit
   on evidence ("non-empty" would then need a per-type definition first).

## 2. Tests

- **Snapshot:** defaults pre-hydrate · hydrate overlays stored overrides · out-of-range
  setEngineParam throws loudly, writes nothing · session-consistency pin (decision 1) ·
  hydrate with empty store ≡ pure defaults (fresh-install determinism).
- **Rejection:** >30%-bad-dates fixture → `ColumnTransformRejection` with exact counts,
  threshold, and ALL cellErrors (count pinned) · <30% → applied with per-cell issues flagged
  (existing behavior re-asserted) · the per-COLUMN boundary pin (decision 2) · threshold read
  from a store override (not the default) changes the verdict — proves store-backing live.
- **Q-009:** UnmappedColumnsError enumerates names/ids · getter matches · the ⟺ pin
  (decision 3).
- **Ported-suite integrity:** the 1,347-line stage2 suite and column suite stay green with
  zero assertion weakening (the rejection subclasses LocalizableException, so existing
  `toThrow(LocalizableException)` pins still hold).
- **Determinism:** no clock/random; suite under hostile TZ as always.
- E2E: pipeline-e2e extended — one fixture column driven over threshold via store override,
  rejected loudly, session completes for the rest.

## DoD (acceptance from the prompt + the pins)

1. Fixture with >30% bad dates → type rejected + reasons listed (the complete cellErrors
   payload); <30% → applied with per-cell issues flagged.
2. Thresholds read from config (store-backed), not UI; store override demonstrably changes
   gate behavior; NFR-009 contract holds at the store API.
3. The three pinned tests green: session consistency · per-column boundary · Q-009 ⟺.
4. Denominator faithful (all rows); ported suites unweakened; clean-install gate green
   (engine 720+new / web 66 / functions 73); boundary/barrel byte-unchanged; no new migration.

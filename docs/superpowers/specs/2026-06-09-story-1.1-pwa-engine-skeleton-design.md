## Goal

Make the **NFR-003 UI/Engine boundary structural, not conventional**, and ship a React+TS+Vite
installable PWA that loads offline — with a trivial end-to-end slice (`ping`/`version`) proving the
async-client contract, and a *verified* seam that lets the engine relocate into a worker later with
**zero UI changes**. No engine internals are ported in 1.1 (that is EP-2/3/4/6 PORT/ADAPT work); the
boundary is shaped to *wrap the prior-art facade* (`abc-engine-impl.ts`) without yet doing so.

## Hard constraints honored

- 🔒 **HC-1 local-only:** no Auth, Analytics, A/B, AI, cloud sync. Firebase Hosting config only.
- 🔒 **HC-2/3 minimization:** no persistence designed in 1.1; no raw-ledger assumptions introduced.
- 🔒 **HC-9 determinism:** no ML/AI.
- **VIS-002 local-first:** all compute/storage in-browser. Nothing leaves the device in 1.1.
- **NFR-005:** Plus-features (AI, backup, bank-sync, auth) live **outside** the Core workspace from
  day 1 — a future private submodule, never a Core package.

## 1. Architecture & boundary model

Two packages in a **pnpm workspace** (not Nx), wired with **TypeScript project references** and built by Vite:

- **`@abc-budget/engine`** — framework-free TypeScript. Public API is *only* the `EngineClient`
  interface, its DTO types, and factory functions. The engine impl, the IoC container, RxJS, and
  (later) Dexie are **internal** — invisible past the package boundary. RxJS is an internal detail,
  adapted to Promises at the client boundary; it never appears in the public surface.
- **`apps/web`** — the React PWA. Depends on `@abc-budget/engine` and imports **nothing but the
  client surface**.

The boundary is enforced **three ways at once**, so it is a build property, not a convention:

1. **Package `exports` map** — only the client entrypoint (`.` → barrel) is resolvable; deep paths
   into `@abc-budget/engine/src/internal/...` **fail to resolve**.
2. **TS project references** — no deep type paths leak across the package edge.
3. **ESLint flat-config `no-restricted-imports`** — backstop forbidding deep engine imports.

**Relocation seam.** The UI only ever holds an `EngineClient`. Whether that client talks to an
in-thread impl or a worker is decided by *which factory the app calls* — the interface is identical,
so moving the engine into a worker is zero-UI-change **by construction**.

## 3. The `EngineClient` contract (engine-shaped, minimal)

Shaped to *grow into* the prior-art facade, but 1.1 ships only the trivial slice:

```ts
// packages/engine/src/client/engine-client.ts  (PUBLIC)
export interface EngineVersion {
  engine: string;    // semver of @abc-budget/engine
  contract: number;  // integer — the client-versioning seam (flagged open)
}

export interface EngineClient {
  /** Echoes the message back — proves serializable args+returns survive a thread hop. */
  ping(message: string): Promise<string>;
  getVersion(): Promise<EngineVersion>;
}
```

Everything is `async` and every argument/return is **structured-clone-serializable**, so the same
interface works whether the impl is in-thread or behind `postMessage`. The prior-art
`databaseName: Observable` becomes a plain setter when the facade is actually wrapped in EP-2; it is
**out of scope for 1.1** — the impl behind the client is a stub `PingEngine`, no facade is ported yet.

## 4. Transport seam & the worker spike

- **`createDirectEngineClient(): EngineClient`** — the app's real wiring. Instantiates the internal
  stub engine and calls it directly in-thread; no marshalling overhead now.
- **`createWorkerEngineClient(worker): EngineClient`** — **spike/test only, not shipped.** Same
  `EngineClient` type, backed by `postMessage`. Its existence proves the contract survives the thread
  hop and that swapping transports is zero-UI-change.

**Wire protocol:** a tiny hand-rolled envelope — `{ id, method, args }` ⇄ `{ id, ok, value | error }`
— with **zero deps**, so we *own* the versioning + transaction-boundary knobs we will need later.
Comlink is a documented alternative, deliberately **not** adopted: we already flagged
transaction-boundary + versioning as open, so owning the protocol beats hiding it behind a library.

The production transport and the real engine host (a **dedicated/Shared Worker**, **not** the offline
SW) are deferred to **EP-2.6**.

## 5. PWA & offline

`vite-plugin-pwa` in **`generateSW`** mode (Workbox) — zero custom SW code. Web-app manifest
(name «ABC Budget», `display: standalone`, icons, theme color), `registerType: 'autoUpdate'`, Workbox
precaches the built app-shell. No custom service worker and **no engine in the offline SW** — the
offline-caching concern and the engine-hosting concern stay decoupled.

## 6. Deploy & offline proof

`firebase.json`: public `apps/web/dist`, SPA rewrite to `/index.html`, `Cache-Control: immutable` for
hashed assets and `no-cache` for `index.html` + the service worker. `.firebaserc` default project →
`abc-budget-2d379`.

Offline is proven **locally**: `pnpm build` → `firebase emulators:start --only hosting` (or
`vite preview`) → toggle DevTools offline → confirm the app-shell still loads and `ping()` works.

The live `firebase deploy` command is **documented but not run** — the founder deploys via their own
`firebase login`. 🔒 The service-account JSON is gitignored (`*service-account*.json`), never enters
the repo, and is **not** used for any manual deploy.

## 7. Testing (verified, not declared)

`vitest`. Three checks:

- **(a) Contract** — `ping`/`getVersion` via `createDirectEngineClient` return the expected values.
- **(b) Worker-hop seam** — build a `WorkerEngineClient` against a **real `Worker`** and assert `ping`
  echoes, proving serializability + the zero-UI-change seam. This **requires a real Worker
  environment**: use **`@vitest/web-worker`** (or vitest browser mode). A plain jsdom run has no
  `Worker` and would *silently* fail to cross a thread — the test must genuinely cross the boundary.
- **(c) Boundary guard** — **structural, enforced by the `exports` map at the module-resolution
  layer.** *(Revised during implementation — see note below.)* The engine's `package.json` publishes
  `exports` for `"."` only, so a deep import into engine internals throws `ERR_PACKAGE_PATH_NOT_EXPORTED`
  under Node's ESM resolver — the same enforcement Vite applies to production builds. The automated
  test (`apps/web/src/__boundary__/boundary.spec.ts`) asserts (1) the public barrel exposes exactly
  `{ createDirectEngineClient }` at runtime (no internals leak) and (2) a deep internal import is
  refused by the resolver. **ESLint `no-restricted-imports`** is the dev-time guard (forbids
  `@abc-budget/engine/*` deep paths in `apps/web`).

  > **Topology note (why not a typecheck guard):** the originally-specified `@ts-expect-error`
  > typecheck fixture is **infeasible** here. Under pnpm workspace symlinks + TS **project
  > references**, `tsc` resolves `@abc-budget/engine/*` through the referenced project's source/
  > declarations, **bypassing the `exports` map** — so the deep import resolves at the type layer and
  > the directive can't fire. The `exports` boundary is real at build/runtime (Vite/Node), just not at
  > `tsc`. Project references are retained for typechecking only. This was approved as a strict
  > improvement: the boundary is enforced by the module system itself, not merely by lint.

## 9. Open items & carry-forward (per NFR-003 "open")

- **Transaction boundary** across the client — undecided; owned by our envelope protocol.
- **Client/contract versioning** — the `contract: number` placeholder is the concrete seam.
- **Final wire protocol** — hand-rolled envelope vs Comlink; revisit at EP-2.6.
- **Real engine host** — dedicated Worker vs SharedWorker; lands EP-2.6.
- **Worker transport hardening (EP-2.6, from the Task 3 review)** — when the spike becomes the
  production transport, add: a `worker.onerror`/`onmessageerror` handler that drains+rejects the
  pending map; a per-request timeout (a never-responding worker currently hangs the caller forever);
  structured error serialization (`String(error)` loses the error class today); and an explicit test
  of the error/reject path (only the happy path is covered now).
- **Boundary at typecheck** — not achievable under pnpm symlinks + TS project references (see §7c
  topology note). If a future TS/workspace change makes `exports` enforceable under `tsc`, add a
  typecheck guard on top of the existing exports-resolution test. Until then the structural boundary
  lives at the module-resolution layer (Vite/Node) plus ESLint.
- **`exports` → raw `./src/index.ts`** is fine for 1.1 (workspace + Vite compile from source). When
  the engine becomes a **worker bundle (EP-2.6)** or is **published as OSS (NFR-005)**, `exports` will
  need a built `dist` + `types`. Keep the barrel clean — do not let source-exports leak build
  assumptions.
- **`registerType: 'autoUpdate'`** is fine for 1.1; the update UX (prompt-to-refresh vs silent
  auto-reload) is revisited later.

## Definition of Done (what the PM verifies)

1. `pnpm install && pnpm build` clean.
2. App installs as a PWA and **loads offline** after first visit (local proof documented: how it was checked).
3. Tests green: **contract** (×2), **worker-hop through a real Worker**, **boundary via exports-resolution** (×2: public-surface + deep-import-refused), **App vertical slice**.
4. `apps/web` sees only the public client — a deep `@abc-budget/engine/...` import is refused by the `exports` map at build/runtime (`ERR_PACKAGE_PATH_NOT_EXPORTED`) and by ESLint at dev-time.
5. No Firebase Auth/Analytics/A-B; Hosting config only; no service-account secret in the repo.

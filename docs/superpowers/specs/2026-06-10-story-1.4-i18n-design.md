## Goal

Chrome-string i18n (uk/en) with auto-locale default, a persistent UK/EN toggle in all three 1.5
headers, localStorage persistence with a **synchronous pre-render read** (no language flash), and
structural protection against user content ever entering translation.

## Locked decisions (founder, pre-made + design round)

1. **Persistence: `localStorage`, key `abc.lang.v1`**, read SYNCHRONOUSLY before first render —
   no flash. (localStorage, not the engine DB — decided at 1.2: the boot read must be sync.)
2. **Default:** `navigator.language` → `uk` if it starts with `uk`, else `en` (local, no network);
   saved override wins over locale. The prototype's own `detectLang`/`persistLang`
   (`dashboard-data.jsx:31-32`) implement exactly this — ported near-verbatim, try/catch-guarded.
3. **Chrome-only translation.** User content (category names/descriptions) never passes through
   translation — and 1.4 creates no place where it could (structural, §4).
4. **Toggle fills the existing `<LangSlot/>`** in all THREE headers (DwellHeader, FlowHeader,
   OnboardingHeader) — reachable from first-run (FEAT-028: a wrong auto-detect must be fixable on
   Onboarding itself, otherwise the onboarding copy is unreadable).
5. **Scope: strings/toggle/persistence only.** Number/currency formatting → 1.6 (ENT-019).
6. **Framework: hand-rolled typed catalog** (~50 lines, zero deps, mirrors the designer's `STR`
   pattern). react-i18next/@lingui rejected: ~20KB + ICU machinery unused by this scope, async-init
   patterns fight the sync-boot requirement, extraction toolchains are EP-scale YAGNI. The catalogs
   port mechanically if ICU-scale needs ever arrive.

## 1. Module layout

```
apps/web/src/
├─ app/i18n/
│  ├─ catalog-uk.ts          # SOURCE OF TRUTH: flat typed chrome catalog (uk)
│  ├─ catalog-en.ts          # en, `satisfies Record<ChromeKey, string>` → key drift = compile error
│  ├─ i18n.ts                # Lang type ('uk'|'en'), detectLang/persistLang (prototype port),
│  │                         #   INITIAL_LANG = detectLang() at MODULE LOAD (sync, pre-render),
│  │                         #   t(lang, key, params?) with tiny {param} interpolation
│  ├─ i18n.spec.ts           # detect priority, persistence, parity, interpolation, @ts-expect-error fixture
│  └─ LangProvider.tsx       # context {lang, setLang}; setLang persists + sets <html lang>;
│                            #   useLang(), useT() → t bound to current lang
├─ ui/altus/components/LangToggle.tsx   # presentational {lang, onChange} — markup verbatim from
│                                       #   prototype dashboard-app.jsx LangToggle (globe + UK/EN,
│                                       #   aria-pressed, role="group")
├─ ui/altus/altus-components.css        # += .langtog/.langbtn/.globe (extracted from Dashboard.html)
├─ ui/altus/gallery/Gallery.tsx         # += LangToggle states (presentational fixture)

```

## 2. No-flash boot

`INITIAL_LANG` is computed at **module evaluation time** — `main.tsx`'s import graph runs it before
`createRoot`, so the first render is already in the detected/persisted language. No effect, no
re-render, no flash. `LangProvider` initializes from `INITIAL_LANG` (tests may inject
`initialLang`); `setLang` persists via `persistLang` and sets `document.documentElement.lang`;
an init statement sets `<html lang>` for the initial value too.

## 3. Strings externalized

All 1.5 chrome strings move to the catalogs: zone labels (Дашборд/Налаштування), Settings section
tabs, stepper labels (ФАЙЛ/КОЛОНКИ/КАТЕГОРІЇ/ОГЛЯД) + «КРОК {n} / {total}» (the `t` interpolation
helper — no library), screen titles/copy, all CTA and wizard key labels.
**Deliberately not localized:** the gallery (QA chrome, hardcoded specimens — founder-locked) and
the Dashboard CRT machine readout (`ENGINE … · CONTRACT … · PING OK` — status codes, not prose).

## 4. User-content protection (HC-6 / VIS-003)

Structural: `useT()` exposes `t(key: ChromeKey, params?)` with `ChromeKey = keyof typeof CATALOG_UK`
— a dynamic/user string cannot enter translation at compile time; there is no `t(string)` escape
hatch. Verified by (a) a `@ts-expect-error` fixture (`t('user typed this')` must not compile) and
(b) a runtime DoD test: a controlled input holding «Мої продукти», language toggled, value asserted
byte-identical.

## 5. Existing-tests nuance

jsdom's `navigator.language` is `en-US` → tests would default to en, but 1.5's 18 router tests
query Ukrainian names. Fix: `router.spec.tsx` renders through `LangProvider initialLang="uk"` (one
wrapper edit; assertions unchanged). New tests: detect priority (`saved > locale > en`: uk-UA→uk,
en-GB→en, de-DE→en; invalid saved value ignored), persistence round-trip, no-flash (first render
already uk, no effect-driven re-render), toggle present+functional in all three headers incl.
Onboarding, catalog key-parity (runtime set-equality + the compile-time `satisfies`),
`<html lang>` sync, user-content invariant.

## 6. Out of scope / carry-forward

Number/currency formatting + currency reference name localization → 1.6 (ENT-019) · engine
untouched (contract = 1) · altus fence intact (LangToggle is props-only — no context/i18n imports
inside altus) · plurals/ICU → only if a real need appears (catalogs port mechanically).

## Definition of Done (what the PM verifies — the ONE checkpoint)

1. `pnpm install && pnpm typecheck && pnpm lint && pnpm test && pnpm build` clean; engine 61
   untouched; all pre-existing web tests green (assertions unmodified; router.spec gains only the
   provider wrapper).
2. PM click-through of the built shell: boot language follows browser locale; toggle present and

4. User-content invariant: toggling language never alters typed input content (runtime test) and

5. `<html lang>` reflects the active language. Fence checks green (no i18n/context imports inside
   `altus/`; LangToggle takes props only).

# Story 3.2 — Dup-counter discriminator (Q-011 second half) — Design

> **Source prompt:** `requirements/deliverables/2026-06-14-EP-3-footprint-dedup-dev-prompts.md` → "## Story 3.2" (6 binding decisions).
> **KB:** Q-011 (the discriminator decision — second half), ENT-012 (hash recipe), ENT-014 (dedup), Q-002/Q-008/Q-010.
> **Flow:** execution → dev final review → STOP, hand to PM → ABC QA (full protocol resumes here).

## Goal

Add the **dup-counter** to the stage3 row hash so two FULL-row duplicates (identical across all 9 recipe fields AND the same `pseudoOp`) get DISTINCT final hashes — `{h#0, h#1, …}` per group of identical base hashes. This is the one piece `hash.ts` deferred (the 9-field recipe + `pseudoOp` marker already exist). Pure stage3 hashing: NO persistence (3.3/3.4), NO contract bump.

## Binding decisions (PM, 2026-06-14)

1. **Batch-deterministic, never against the store.** Group the import batch's rows by base hash (full recipe incl. `pseudoOp`); within each group assign `0..n-1`. Counting against the persisted store would make a re-import see N and assign N..2N-1 → silent double-count. Batch counting → the same statement always yields the same SET `{h#0..h#n-1}` → idempotent upsert in 3.4.
2. **Stable by COUNT, not order** (HC-9). Identical rows are unorderable; assigning `{0..n-1}` to a group of n yields the same final-hash SET regardless of row order. **PIN:** shuffle the batch's rows → byte-identical SET of output hashes.
3. **No fragile inter-row links.** The counter falls out of a pure group-by-base-hash over the batch — no positional/sequence coupling.
4. **Overlapping-range stability.** Bank exports are by whole date, so any range covering an op's day includes ALL its same-day occurrences → the per-group identical count is stable across widening ranges. **PIN:** a fixture with an in-month duplicate, generated over Jan–Mar then Jan–Dec → the dup pair's `{h#0,h#1}` set is reproduced (3.4 proves no double-count on upsert; 3.2 proves the hashes are reproducible).
5. **Counter IN the hash, not a separate column** (ENT-014): a separate counter column under the `(hash,year,month)` unique key would let the index merge true duplicates; in the hash, the duplicates are legitimately-different records the index won't reject.
6. **Mechanism = wrap → re-SHA (opaque 64-hex)** (PM ruling): `finalHash = SHA-256({ base: <baseHash>, dup: k })`, where `baseHash` = the EXISTING 9-field+`pseudoOp` recipe, left BYTE-UNCHANGED. Serves decision 5 with a real SHA — the stored hash stays a uniform opaque 64-hex with nothing parseable (rejected: `baseHash#k` leaks splittable structure vs ENT-001/VIS-002; embedding `dup` in the recipe object re-introduces the "undocumented extra key" confusion + couples the batch counter into the per-row recipe fn).

## 1. Design

**`hash.ts` — the base recipe is untouched.** `generateHashableObject` / `calculateRowHash` (9 fields + `pseudoOp`) stay byte-identical — they produce the **base hash**. The dup-counter is a NEW, clearly-separate batch layer:

```ts
// hash.ts (new)
// Batch post-pass: given the base hashes of every generated row IN ORDER, return
// the final hashes (base wrapped with the per-group dup counter). Pure; deterministic.
export async function applyDupCounters(baseHashes: readonly string[]): Promise<string[]>
//   group indices by baseHash; within each group assign k = 0,1,2… in index order;
//   finalHash[i] = objectHash({ base: baseHashes[i], dup: k })
```

- The group key is the base hash string (same base object → same base hash; SHA-256 collision is negligible).
- `k` is assigned in iteration order, but because the grouped rows are identical, the resulting SET of final hashes is invariant under input reordering (decision 2). Per-index assignment may differ across orders; the SET does not — the PIN asserts the SET.
- Singletons → `k = 0` → `objectHash({ base, dup: 0 })`. Every row's final hash is thus a wrapped SHA (the bare base hash is never the stored value).

**`row-generator.ts` — a batch post-pass.** `generateSingleRow` and `pseudo-ops.ts` continue to set `row.hash` to the **base** hash (via `calculateRowHash`, unchanged). After the generation loop AND the existing re-index pass, `generateRows` runs:

```ts
const finals = await applyDupCounters(resultRows.map((r) => r.hash));
resultRows.forEach((r, i) => { r.hash = finals[i]; });
```

This is the only row-generator change — one post-pass over the final `resultRows` (mains + commission + cashback pseudo-ops together = the batch). Skipped/errored rows produce no `TransactionRow`, so they're correctly absent from the grouping. Grouping over the full recipe incl. `pseudoOp` means a main and its commission pseudo-op are in different groups (already distinct) — only genuine full-row duplicates of the SAME kind collide and get suffixed.

**No DTO/contract change:** `TransactionRow.hash` is still a string; `GenerateResultDTO` serializes it as before. `CONTRACT_VERSION` frozen at 3.

## 2. Tests (TDD — decisions 1–6 as tests)

- **`applyDupCounters` unit (hash.spec.ts or a new dup-counter.spec.ts):** one base hash → one final (dup 0); two identical base hashes → two DISTINCT finals; three identical → three distinct; distinct base hashes → distinct finals (each dup 0); decision-6 shape (final = `objectHash({base, dup:k})` — assert against a direct compute).
- **Shuffle-stability PIN (decision 2, HC-9):** a batch with a duplicate group; compute finals for the batch and for a shuffled permutation → the SORTED sets are byte-identical. (Per-index assignment may differ; the set is invariant.)
- **No-store / batch-deterministic (decision 1):** generating the same batch twice (fresh) → identical final-hash sets (no hidden running state across calls); two separate single-row batches each yield `dup 0` (not 0 then 1) — proves counting is per-batch, not global.
- **Overlap PIN (decision 4):** a fixture with an in-month duplicate; `generateRows` over the Jan–Mar subset and over the Jan–Dec superset → the duplicate pair's final-hash set `{h#0,h#1}` is identical across both (both occurrences present in both ranges).
- **End-to-end via generateRows:** a stage2 batch with two identical mappable rows → the two generated rows have DISTINCT hashes; a single occurrence → its own hash; pseudo-ops already distinct via the marker (re-assert one main+commission pair stays distinct, now also dup-wrapped).
- **Ported-suite integrity:** hash.spec's pseudoOp/COUNTERPARTY base-recipe pins stay green (base recipe untouched); pipeline-e2e / real-statements determinism + counts unchanged (absolute hash values change — wrapped — but nothing baked depends on them; run-to-run determinism holds).
- No clock/random; engine suite under hostile TZ as always.

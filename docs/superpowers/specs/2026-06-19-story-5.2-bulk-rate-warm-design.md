### The bulk flow (the founder's plan — ONE round-trip each)
1. **Guards** (mirror the order): method(POST) → `checkOrigin`(403) → `rateLimiterTake(ip)` **1 token for the
   bulk request** (429) → validate the list: each `validateDate`+`isDateInRange` (400 on any invalid), a
   `MAX_BULK_DATES` cap (400 if exceeded — bounds getAll + OER volume), non-empty (400). De-dup the dates.
2. **ONE `db.getAll(...dateRefs)`** over the `usd/{date}` refs → split HITS (return verbatim) / MISSES.
3. **ONE cap-transaction** on `meta/oer-budget`: `checkAndIncrementByK(docData, key, cap, want=missCount)` →
   `allowed = min(want, cap - current)`; write `current + allowed`. Returns `allowed` (0..want). Over-cap →
   `allowed = remaining` (maybe 0). We fetch only `allowed` of the misses (deterministic: misses sorted by date,
   take the first `allowed`); the rest are OMITTED (best-effort). **One txn; increment upfront by `allowed`** (a
   rare failed OER fetch is still counted — the soft-cap trade-off; noted).
4. **Bounded-concurrency OER fetch** of the `allowed` misses — a small pool (`CONCURRENCY≈5`): NOT N sequential
   (CF timeout) nor N parallel (OER rate-limit). A per-date OER failure → OMIT that date (best-effort), don't abort.
5. **ONE `WriteBatch`** (chunk ≤500 ops) writing the fetched daily tables to `usd/{date}`.
6. **Return** the merged `{ rates: { [date]: table } }` (hits + fetched). Omitted dates simply absent.

### `budget.ts` increment-by-K [FORK — recommend a new pure fn alongside the +1]
```ts
export function checkAndIncrementByK(docData, key, cap, want): { allowed: number; next: Record<string, number> } {
  const current = docData[key] ?? 0;
  const allowed = Math.max(0, Math.min(want, cap - current));   // over-cap → allowed = remaining (maybe 0)
  return { allowed, next: allowed > 0 ? { ...docData, [key]: current + allowed } : { ...docData } };
}
```
Run inside ONE `db.runTransaction` (atomic; Firestore serializes racing bulk requests). The +1
`checkAndIncrementMonthlyCap` stays for the per-date endpoint.

### Chunking + caps
`getAll` ≤3000 (a statement's distinct dates ≪ that — no chunk needed; `MAX_BULK_DATES≈366` bounds it).
`WriteBatch` ≤500 ops → chunk the fetched misses into ≤500-op batches (usually one). `CONCURRENCY≈5`,
`MAX_BULK_DATES≈366` (one year) — confirm the constants.

### Privacy / logging
The bulk handler logs SUMMARY COUNTS ONLY — `{ requested, cached, fetched, saved, capCut }` — **never the dates
list** (the per-request-date-logging hygiene). The per-date `getUSDRates` keeps its existing logging.

## The S3b→S3c trigger [FORK — recommend importNext + fire-and-forget]

  rejected (EP-2's offline-import promise + slower transition).*

## best-effort semantics
The bulk CF returns a PARTIAL map (cap-cut / OER-failed dates omitted). The worker writes what it got. Uncached
dates surface ONLY at commit via `CacheOnlyRatesApi` → `RatesUnavailableError` (the loud gate, retry-able). The
warm NEVER errors the import (S3c magnitude is fail-soft; the commit is the single guarantee).

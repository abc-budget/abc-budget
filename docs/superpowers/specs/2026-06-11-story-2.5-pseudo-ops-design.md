# Story 2.5 — Pseudo-ops expansion (BANK_COMMISSION / CASHBACK) — Design

> **Epic:** EP-2 · **Story:** 2.5 · **Date:** 2026-06-11 · **Branch:** `story/2.5-pseudo-ops` (off `main`, post #12)
> **Source prompts:** `requirements/deliverables/2026-06-10-EP-2-ingestion-dev-prompts.md` (Story 2.5)
> **KB:** ENT-013 (expansion mechanism), ENT-006 (row fields; isBankCommission/isCashback are
> flags of DERIVED ops), ENT-011 (CurrencyDetectOptions), ENT-009 (column types), Q-011
> (RESOLVED-direction: hash discriminator), FEAT-022 (collect-don't-throw), VIS-011.
> **Flow:** execution → dev final review (with PM checkpoint (b) evidence) → STOP, hand back
> to founder (role fence).

## Goal

A statement row with non-empty BANK_COMMISSION and/or CASHBACK cells spawns separate
pseudo-operations (date+account from the original row; amount+currency from the respective
column; flag + synthetic description). The 2.3 `{rows, rowErrors, skipped}` never-throw
contract holds for pseudo-ops. Last engine story before PM checkpoint (b) — the final review
carries full-pipeline evidence (fixtures + real statements, including pseudo-op counts).

## Current state

- Stage2 (2.2 port) already parses both columns: empty cells (`—` etc.) → `value: null`;
  numbers → plain values; each column carries its own ENT-011 currency params.
- The 2.3 row-generator ignores both columns; `isBankCommission`/`isCashback` hardcoded false.
- mono UA real statement: 27/181 rows carry cashback (plain positive decimals), 0 commission.

## Locked decisions (founder, 2026-06-11)

1. **`+`-prefix: FIX parseNumber (declared divergence from prior art).** ONE RULE, ALL
   NUMERIC PATHS: `+` accepted wherever `-` already is — at the parseNumber level, never a
   CASHBACK special case. Pinned with «+5000,00». **Sign semantics unchanged:** `+` is
   notation for positive — «+5000,00» in a MIXED column is still the income side →
   skipped-with-reason (the composition is pinned with one test). Rationale on record:
   composed with the 2.4 gate, loud-NaN = column LOCKOUT for `+`-writing banks (not noise) —
   the wedge anti-pattern; the real set (mono) is unaffected either way.
2. **Q-011 scope: type-marker only.** The pseudo-op discriminator (`main | commission |
   cashback`) enters the canonical hash object NOW. Pins: one source row with
   commission+cashback → THREE distinct hashes; same-type pseudo-ops from different rows
   keep hashing by their own fields. The dup-counter (identical full rows → suffix 0,1,2…)
   carries to EP-3 EXPLICITLY — ledger tag: "Q-011 second half: pin 'stable count, not
   order' against the real upsert path". Hash recipe is still persistence-free.
3. **Spawn scope: generated AND income-skipped mains; NEVER errored rows.** The decisive
   case: an income transfer (+50 000) with a 100₴ commission — the main is skipped (VIS-011,
   correct), but the commission is REAL EXPENSE; dropping it would silently lose money (the
   worst failure class). Pseudo-ops are independent operations per ENT-013 — their nature
   does not inherit the main's direction. Errored rows never spawn (no reliable
   date/account to donate; the rowError already reports loudly — nothing silent). Pins:
   (1) income-skipped main + commission cell → main in `skipped` WITH reason AND exactly one
   commission pseudo-op in `rows`; same for cashback; both cells → 2 pseudo-ops + skipped
   main. (2) Errored row + non-empty commission cell → rowError only, ZERO pseudo-ops.
   (3) The income+commission fixture enters the matrix README (QA matrix source).

## 1. Pseudo-op shape (convention-derived, stated for the record)

For each non-empty BANK_COMMISSION / CASHBACK cell of an eligible row (decision 3):

| Field | Value |
|---|---|
| `date`, `account`, `source` | from the original row |
| `amount` | abs value from the respective cell (flags carry semantics — consistent with main-op convention; cashback's income-ness lives in `isCashback`, not the sign) |
| `currency` | per the COLUMN's ENT-011 CurrencyDetectOptions (auto / use_base / {code}), resolved with the same machinery as AMOUNT |
| `description` | synthetic catalog key via the 2.4 `$t` convention: `engine.importStatement.pseudo-op.bank-commission` («Комісія банку») / `.cashback` («Кешбек») — uk/en strings land at 2.8; key string is hash-stable |
| `counterparty`, `bankCategory`, `mcc` | `null` |
| `isBankCommission` / `isCashback` | `true` for the respective kind, `false` otherwise |
| `category`, `isManuallySetCategory` | `null` / `false` (EP-4) |
| `hash` | canonical object + the Q-011 discriminator (decision 2); main ops carry `'main'` — the recipe change touches all hashes, still free (HC-2/3: nothing persisted) |
| `rowIndex` | pseudo-ops append after their source row's main op in source order (main, commission, cashback); the existing final re-index pass then syncs rowIndex = array index |

A FAILED pseudo-op (e.g. cell carries a stage2 parse error, currency resolution throws) →
`rowErrors` entry naming the rowIndex and the pseudo-op kind; the MAIN op (and the sibling
pseudo-op) are unaffected — FEAT-022 per-operation granularity, generation continues.

# Story 4.9a.1 — Recalled mappings don't parse their cells — ROOT-CAUSE NOTE

> BLOCKER (QA-found on 4.9a; root cause is a latent EP-2 recall defect on MAIN). ROOT-CAUSE NOTE FIRST —
> confirm the seam + the source-fix plan, STOP; PM reviews before the fix (the FIX LAYER is the risk).
> Branch `story/4.9a.1-recall-cell-parse` off `main` @ `797cfc2`.

## The symptom
Re-import a previously-mapped bank (columns recalled from the pool as `guessed`) → advance to categorization
→ `row.date.getUTCFullYear is not a function`. `importGetRows[0].date = "30.09.2023 14:55:43"` — the RAW cell
STRING, not a `Date`.

## The confirmed root cause (file:line + evidence)
**The recall MOUNT sets the column DEFINITION but never runs the cell PARSE.** Two seams:
1. **`stage2/implementation.ts:226-235` — the `applyRecall` closure** (stage2 constructor, recall mount):
   ```ts
   const applyRecall = (col) => {
     const prefill = recallPrefills.get(key);
     if (!prefill) return col;
     return col.copy({ definition: prefill.definition, params: prefill.params, recallState: prefill.state }); // 'guessed'
   };
   ```
   `copy({ definition, params, recallState })` sets the DEFINITION + params + `guessed` state but does **NOT**
   transform the cells. The column's cells stay the **raw decoded strings**.
2. **`column.ts:426` `parseAsDate` (and `parseAsAmount` etc.) is the ONLY thing that produces typed cells** —
   its `parseFunction` maps `cell.value → DateTime.fromFormat(...).toJSDate()` (a real `Date`,
   `type: DATE`). The recall path never calls it.
3. **The dispatcher `direct-client.ts:164-189`** maps `definition → col.parseAsX(params)` — but it is invoked
   **ONLY by `importApplyColumn`** (the INTERACTIVE map). Recalled columns are `guessed`; the user advances
   (or `importConfirmRecall` — a separate method that only transitions `guessed→confirmed` state, see
   `implementation.ts:325` `applyColumn`: state transition + pool write, **no parse**). So a recalled column
   **never goes through the dispatcher → its cells are never parsed**.
4. **`row-generator.ts:345` `extractDate`** then does `row.get(id).value as Date` — a cast that LIES at
   runtime (the value is the raw string). `autoCategorize → distinctPeriods → footprintYearMonth →
   row.date.getUTCFullYear()` is the first consumer to call a `Date` method → crash.

**Systematic-debugging evidence (matches QA's 2×2):** the defect is in the recall **mount** (the constructor
`copy()`), which is **transport-agnostic** (runs in both the direct and worker clients → "not
worker-specific") and **independent of `importNext`** (the cells are unparsed from construction, not at
advance → "not importNext-specific"). It is **triggered by the recall path** (interactive mapping runs
`parseAsX` via `importApplyColumn` → typed cells; recall does not). **Latent on main:** every downstream
consumer read `.value as Date` (the lying cast) and the interactive path always produced real Dates, so the
unparsed-recall cells weren't exercised through a `Date`-method consumer until EP-3/4 (`deriveFootprint` /
`autoCategorize`) existed. 4.9a's `autoCategorize` is simply the first to hit it; EP-5's footprint commit
would hit it identically.

## Scope — NOT date-only (HOLISTIC, confirmed)
The `applyRecall` `copy()` sets the definition for **any** recalled typed column without parsing → **every**
typed column is affected:
- **DATE** → cells stay strings (the reported crash).
- **AMOUNT** → `parseAsAmount` (sign/decimal/`use_base` normalization → a `number`) never runs → `row.amount`
  is a raw string. `autoCategorize` / `remainder-magnitude` / `deriveFootprint` would get a string amount →
  silent wrong math or a crash. **Equally broken on recall, just not yet the surfacing consumer.**
- CURRENCY / DESCRIPTION / COUNTERPARTY / BANK_CATEGORY / BALANCE → their `parseAsX` transforms are also
  skipped (text transforms are ~identity so low-harm; AMOUNT + DATE are the load-bearing ones).
→ The fix MUST be holistic (run the parse for ALL recalled typed columns), not a DATE special-case.

## The source fix (the right layer — NOT `extractDate`/`autoCategorize`)
**Principle:** a recalled-and-applied mapping must run the SAME cell-parse as an interactively-applied one.
Do NOT defensively coerce in `extractDate`/`autoCategorize` — that hides the contract break and every
consumer (EP-5/6) would need the same patch; the stage3 `date: Date` / typed-`amount` contract MUST hold by
the time rows exist.

**Plan:**
1. **Extract the dispatcher.** Lift the `definition → col.parseAsX(params)` switch (`direct-client.ts:164-189`)
   into ONE shared engine helper (e.g. `parseColumnByDefinition(col, definition, params)` on stage2 / a
   stage2 util). Both the interactive path AND the recall path call it → ONE source of truth, no drift.
2. **Run the parse for recalled columns at the recall-application layer.** The constructor `applyRecall` is
   SYNC and `parseAsX` is async (lazy luxon) → the parse runs in **`service.startWith`** (the async session
   bootstrap that already does `recallFor` + constructs stage2): after construction, for EACH prefilled
   column, `await parseColumnByDefinition(col, prefill.definition, prefill.params)` so the recalled column
   gets **typed cells** — identical to an interactively-applied column.
3. **Preserve `recallState: 'guessed'`** (the FIX-LAYER subtlety to confirm): the recalled column must keep
   its `guessed` state (awaiting the user's endorsement/advance) WHILE having typed cells. But the parse path
   (`parseAsX → applyColumn`) transitions `guessed→confirmed`. → the recall-mount parse must **preserve
   `guessed`** (re-stamp `guessed` after the parse, or a parse variant that does not transition). This is the
   one nuance the PM flagged — **recommend: run the parse, then `applyColumn(col.copy({ recallState: 'guessed' }))`
   to restore the awaiting-confirm state** (the cells are typed; the endorsement still happens at advance via
   the existing `flushRecallWrites`). Confirm this vs. a `keepRecallState` parse flag.
   - *Rejected layer:* parsing at `importConfirmRecall`/advance only — the user can ADVANCE with `guessed`
     columns (advance = endorse), so a confirm-only parse would still leave an advance-without-explicit-confirm
     path unparsed. Mount-time parse is robust to both.

## 1. The sandbox session SHAPE (headless) — `rules/rule-sandbox.ts`
`class RuleSandboxSession`, constructed once per import editing session with everything pre-loaded (zero DB
in the hot path):
```ts
new RuleSandboxSession({
  importRows: ImportStatementStage3Row[],          // the current import's in-memory rows (~1k–10k)
  overrideMap: Map<string,string>,                 // 4.4 — FIXED for the session
  categoriesById: Map<string,Category>,            // 4.4 — FIXED for the session
  currentTree: DecisionTree,                        // 4.3b reload() — the persisted tree
  persistence: RulePersistenceService,             // 4.3b — for live writes + apply
})
```
State: `currentTree`, `virtualTree: DecisionTree | null` (null ⇒ not engaged), `engaged: boolean`.
Lifecycle:
- **Not engaged + LIVE action** → mutate `currentTree` + persist immediately (the "update" path).
- **Not engaged + SANDBOX action** → ENGAGE: seed `virtualTree` from `currentTree` + the action; enter review.
- **Engaged + ANY action** (incl. append) → ACCUMULATE into `virtualTree` (FEAT-029: once in sandbox, every
  subsequent edit, even an append-at-end, accumulates — no second live path while engaged).
- **`apply()`** → `persistence.saveDecisionTree(virtualTree)` (4.3b delta → matched rules keep ids, minimal
  store churn) → `currentTree = virtualTree`; `virtualTree = null`; disengage.
- **`cancel()`** → `virtualTree = null`; `currentTree` unchanged; disengage.
- **`diff()`** → the changed-rows result (below). All edits go through the builders (`ComplexRuleBuilder`/
  `DecisionTreeBuilder`) to construct the new tree — never hand-mutate.

**Edit triggers ONLY on save.** Opening a rule (loading its conditions into the constructor as a
filter-preview) is a 4.9 UI concern — 4.5 models only save-time actions; `currentTree`/`virtualTree` are
untouched by "open".

## 2. The structural-trigger classification (O(1), by action TYPE, NEVER from the diff)
```ts
type EditAction =
  | { kind: 'reorder'; order: number[] }
  | { kind: 'delete'; ruleId: number }
  | { kind: 'editConditions'; ruleId: number; before: Rule[]; after: Rule[] }
  | { kind: 'categoryOnly'; ruleId: number; categoryId: string }
  | { kind: 'appendEnd'; rule: ComplexRule };

function classify(action): 'sandbox' | 'live'
  // reorder → sandbox · delete → sandbox · categoryOnly → live · appendEnd → live
  // editConditions → canonicalEqual(before, after) ? 'live'/*no-op*/ : 'sandbox'
```
- Pure, O(1) by `kind` (the `editConditions` branch is O(#conditions) for the canonical compare — bounded,
  NOT O(diff)). **The trigger NEVER consults the diff** (a delete that happens to change 0 rows still
  sandboxes — consistency over cleverness; no "last rule" exception).
- The classification IS the **dynamic-label decision**: `live` → "update"; `sandbox` → "review". 4.5 returns
  the decision; 4.9 renders the button label.

## 3. The diff — `resolveCategory` swap-tree (the override-independence proof)
```ts
diff(): { rowIndex; oldCategoryId: string|null; newCategoryId: string|null }[]
  for each importRow:
    const oldC = resolveCategory(row, { overrideMap, categoriesById, tree: currentTree });
    const newC = resolveCategory(row, { overrideMap, categoriesById, tree: virtualTree ?? currentTree });
    if ((oldC?.id ?? null) !== (newC?.id ?? null)) emit { rowIndex, old, new };
```
- Computed via the **4.4 orchestrator**, swapping ONLY `tree` (overrideMap + categoriesById held fixed) —
  NOT bare `tree.categorize`.
- **Override-independence (structural, must be tested):** a row resolved at L1 (`isManuallySetCategory`) or
  L2 (`overrideMap.get(hash)`) returns BEFORE reaching L3 (the tree) → its result is identical under
  `currentTree` and `virtualTree` → it can NEVER differ → it can NEVER appear in the diff. So flag+delete a
  rule that an overridden op matched: the override op is ABSENT from the delta (the rule change is invisible
  to it). This is *why* the diff must go through `resolveCategory`, not `categorize`.
- **ZERO DB per recompute:** the override maps are in-memory + the evaluator is synchronous → `diff()` does no
  DAO reads. Recompute on every accumulated edit is cheap (O(rows × tree), in-memory).

## 4. Canonical condition-equality (the no-op detector) — `rules/canonical-conditions.ts`
- `canonicalConditionSignature(rules: Rule[]): string` — per condition, a canonical key
  `${field}|${operation.type}|${stableSerialize(operation)}`; **sort the keys**, join. Two condition sets are
  equal iff signatures match — **order-INDEPENDENT** (conditions are AND-combined within a complexRule, so
  order is NOT semantic). `stableSerialize` sorts operation keys so `{type,value}` vs `{value,type}` match.
- **Why NOT reuse 4.3b `rulesContentMatch`:** that matcher is **POSITIONAL** —
  `a.rules[i].field !== b.rules[i].field` by index. It exists to reuse a store row's id in the
  `saveDecisionTree` delta, where matching by position is fine. But the no-op detector must treat
  **reordered conditions as equal** (semantically identical AND-set); `rulesContentMatch` would falsely
  report "changed" on a reorder. Distinct purpose → a separate, sort-first helper. (Keep `rulesContentMatch`
  as-is; do not entangle.)
- Drives `classify('editConditions')`: same signature → no-op → `live` (sandbox does NOT engage);
  changed → `sandbox`. **Returning to the SAME condition set (any order) = no sandbox.**

## 5. Apply / Cancel (already in §1)
APPLY = `saveDecisionTree(virtualTree)` (delta-ID-preservation → matched rules keep ids, no complexRules
churn) then virtual becomes current. CANCEL = discard virtual, current unchanged.

## Design decisions for PM confirmation
1. **Session ownership of LIVE writes:** a not-engaged LIVE action (categoryOnly / appendEnd) mutates
   `currentTree` AND persists immediately via `persistence` (create/update); SANDBOX actions persist only on
   `apply()`. Confirm the session owns both (vs. exposing live actions for the caller to persist).
2. **Override ctx fixed for the session:** `overrideMap`/`categoriesById` loaded once (4.4) at session start,
   immutable for its lifetime. Confirm.
3. **Save-time only:** 4.5 models only save-time `EditAction`s; "open a rule" (filter-preview) is 4.9. Confirm.
4. **`CONTRACT_VERSION` frozen at 3** (engine-internal session; the UI wire surface, if any, is 4.9). Confirm.

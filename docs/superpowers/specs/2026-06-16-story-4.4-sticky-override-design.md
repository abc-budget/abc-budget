## The two-level precedence (the model)
A categorized row's category is resolved by, in order:
1. **Level-1 ‚Äî in-session flag:** `row.isManuallySetCategory && row.category` ‚Üí `row.category`.
2. **Level-2 ‚Äî persisted override:** an in-memory `Map<hash, categoryId>` loaded ONCE at import start
   (footprints where `isManual=true`). Keyed on the **EP-3 canonical dup-wrapped hash** ‚Äî format-stable
   across re-imports (3.2 dup-counter is batch-deterministic ‚Üí same statement ‚Üí same hash), so the override
   re-attaches to the same op on re-import.
3. **Level-3 ‚Äî rules:** the 4.2 first-match evaluator over the 4.3b-reloaded tree.
4. **Level-4 ‚Äî** `null`.

- **Why a layer, not a param:** the 4.2 evaluator is the RULES engine ‚Äî keeping override knowledge out of it

  precedence; `categorizeRow` is unchanged.
- **Stays synchronous** because BOTH maps are pre-loaded (no DB in the hot path). `categoriesById` is built
  once at import start (categories service `getAll` ‚Üí ¬´base¬ª resolved). Dangling override id (category
  hard-deleted ‚Äî impossible today, archive-only) ‚Üí log + fall through to rules (mirrors 4.3b RULING 1).
- *Alt rejected:* `categorizeRow(row, overrideMap?)` ‚Äî couples the pure evaluator to persistence concerns.

## üî± FORK B ‚Äî the `isManual` index reality  [recommend: load-once scan+filter; rule the index]
A **boolean keyPath index does NOT function as an IDB lookup**: per the IndexedDB spec a boolean is an
**invalid key**, so records whose `isManual` evaluates to a boolean are simply **excluded from the index** ‚Äî
querying `index('isManual')` returns nothing. **4.3a hit exactly this** with `isArchived` (‚Üí `list()` +
filter; the index was kept inert "for forward use").
- **Recommendation:** the override map LOAD is a **load-once `getAll()` + filter `isManual===true`** (one
  scan at import start ‚Äî satisfies "loaded ONCE, NO per-op query"). For the *index* itself, **please rule**:
  - (A) **keep `isManual` as an inert forward-use index** (consistent with 4.3a `isArchived`) ‚Äî honest that
    it's a placeholder, load stays scan+filter; **OR**
  - (B) **make it genuinely indexable** ‚Äî store the marker as an indexable value (e.g. a sparse
    `manualCategoryId` present only when manual, or `isManual` as `0|1`) and load via that index (useful if
    the footprint store grows to many years of ops). This changes the v8 shape.
  - I lean **(A)** (YAGNI + 4.3a precedent; load-once scan is cheap), but flag (B) for scale. The acceptance
    ("loaded once, no per-op query") is met either way.

## Reset ¬´–ó–∞ –ø—Ä–∞–≤–∏–ª–∞–º–∏¬ª (FEAT-020) ‚Äî what it mutates
`resetToRules(row, ctx)` clears BOTH levels so the row re-evaluates via rules:
1. **Level-1:** `row.isManuallySetCategory = false`, `row.category = null` (clear the in-session flag).
2. **Level-2:** `ctx.overrideMap.delete(row.hash)` (drop the persisted override from the in-memory map).
3. **Next-commit:** mark so `deriveFootprint` writes `isManual=false` + `categoryId = <rule result>` (the
   footprint row is overwritten on the next EP-5 commit via the native `['hash','year','month']` upsert ‚Äî
   no separate delete needed). After reset, `resolveCategory` falls straight through to Level-3 (rules).
- 4.4 ships `resetToRules` (mutates the in-session row + the map); EP-5's commit persists the cleared state.

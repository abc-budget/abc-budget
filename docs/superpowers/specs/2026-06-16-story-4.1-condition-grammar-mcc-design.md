## ADAPT (per ENT-010)
**1. Field-set = ENT-010 exactly:** date, amount, currency, description, counterparty, account,
bankCategory, mcc, isBankCommission, isCashback.
- **+counterparty:** add `createCounterpartyRule(op: StringOperation)` (mirrors description; the
  EP-2 stage3 row already carries `counterparty`). `createStringOperationRule`'s field type widens
  to `'description' | 'account' | 'counterparty'`.
- **в€’source (DROP вЂ” atavism):** delete `createSourceRule`, AND remove `source` from
  `ImportStatementStage3Row` + its construction sites (`row-generator.ts`, `pseudo-ops.ts`, type).
  `source` is `null` since the 2.6 excision and is **not** in the hash recipe
  (`HASH_COLUMN_DEFINITIONS` = DATE/AMOUNT/DESCRIPTION/COUNTERPARTY/CURRENCY/BALANCE/BANK_ACCOUNT/
  MERCHANT_CATEGORY/EXCHANGE_RATE). **Blast-radius pin:** after removal, the hash + dedup specs
  are byte-unchanged (run them; no literal-hash assertion changes). Flagged because it touches
  `importStatement/stage3` outside the new module.

**2. mcc в†’ CATEGORICAL (not numeric range):** `createMccRule(op: StringMatchOperation)` =
`createStringMatchRule('mcc', op)` вЂ” equals / notEquals / oneOf on `String(mcc)`. (Prior-art already
does this; 4.1 **pins** it вЂ” a test asserts mcc takes `StringMatchOperation`, NOT `NumberOperation`
range.)

**3. amountв†”currency pairing вЂ” construction-level [FORK B вЂ” recommend]:** an amount condition is
meaningless without a currency (100 UAH в‰  100 USD). **Recommendation:** export
`createAmountCondition(op: NumberOperation, currency: string): Rule[]` returning
`[amountRule, createCurrencyRule({ type: 'equals', value: currency })]`; keep the bare
`createAmountRule` **un-exported**. Enforcement is construction-level: `currency` is a required
param (won't type-check without it) + a runtime guard throws on empty/blank. A bare amount
condition is thus unconstructable.
- *Alt (rejected):* a single composite Rule (`field:'amount'`) that also checks currency вЂ” muddies
  the single-field grammar and the future ComplexRule AND-composition.

**4. ReDoS guard on `matches(RegExp)` [FORK C вЂ” recommend]:** pure JS can't reliably timebox a
synchronous regex, so the achievable guard is **reject-at-construction + bounded input**:
- `assertSafeRegex(pattern)` (vendored heuristic, NO new dep): reject catastrophic shapes вЂ” nested
  quantifiers `(x+)+`/`(x*)*`/`(x+)*`, quantified groups containing quantifiers, overlapping
  alternation under a quantifier. Called by the string factories when `op.type === 'matches'`;
  throws a typed `UnsafeRegexError` on rejection.
- evaluation caps the tested string length (`MAX_MATCH_INPUT`, e.g. 1000 chars вЂ” bank text is short)
  as defense-in-depth so even an un-caught pattern can't run on a huge input.
- *Dep question:* vendor the heuristic (recommended вЂ” NFR-005 supply-chain minimalism) **vs** add
  `safe-regex` (battle-tested, +1 runtime dep). **Please rule.**

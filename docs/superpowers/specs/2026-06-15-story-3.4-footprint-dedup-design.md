## 1. The `hash` non-unique lookup index — migration **v5 (append)**

**Decision: a new append-only v5 step**, NOT a fold-in to v4.
- v4's `createStore('footprint', …)` is **byte-unchanged** (its own comment: *"the `hash`
  index is deferred to Story 3.4 — do NOT add it here"*). Folding the index into v4 would
  change v4's bytes and break the append-only / fresh≡upgrade discipline every prior story
  honored. The "fold-in if cleaner" alternative you offered conflicts with "v1–v4
  byte-unchanged" — so v5-append is the disciplined resolution.
- The migration `ctx` already supports `createIndex(storeName, indexSpec)` on the upgrade
  transaction (no `rebuildStore` needed — the store already exists). v5 is one line:
  `ctx.createIndex('footprint', { name: 'hash', keyPath: 'hash', options: { unique: false } })`.
- **Pins:** fresh install runs v1→v5 and ends with the index; an upgrade from a v4 DB runs
  only v5 and adds the index → **fresh ≡ upgrade re-proven**; v1–v4 byte-unchanged; the
  `footprint` keyPath `['hash','year','month']` and all prior stores intact.

**Why an index at all (native upsert needs no lookup):** the compound keyPath makes dedup
native — `put()` overwrites by key, no read-before-write. The `hash` index is the
**forward-looking lookup** ENT-014 (evolved) calls for: "find the footprint(s) for this
hash without knowing year/month" (EP-4/EP-6 consumers). 3.4 ships it + a focused
`FootprintDao.findByHash(hash)` (via the base `findByIndex`) + a test. Declared
**non-unique** for safety even though each op's final (dup-wrapped) hash maps to exactly
one row.

## 2. The upsert BATCH write path (engine-internal)

**New module:** `footprint/commit-footprints.ts` — `commitFootprints(rows, deps)`, where
`rows` are stage-3 generated rows that **already carry their final dup-wrapped hash**
(`applyDupCounters` runs in `row-generator.ts:224`, upstream of commit — the write path
does **not** re-hash).

**Orchestration shape — `warm → (convert + derive ALL) → putBatch`, two-phase / all-or-nothing:**
1. **warm** `warmRates(distinctOpDates)` — best-effort, the ONLY network touch. Offline →
   it can't populate; that does not write anything. (Network/HTTP failure here is swallowed;
   the cache-only convert below is the single canonical loud gate, so the failure TYPE is
   uniform regardless of *why* a rate is missing.)
2. **pre-flight convert + derive — pure, NO writes:** for every row,
   `toAmountUSD(amount, currency, opDate)` (cache-only) → `deriveFootprint(row, amountUSD)`.
   A single uncached rate throws **`RatesUnavailableError`** → the **whole commit aborts
   with ZERO writes** (offline+uncached blocks the commit — cleanly, not partially).
3. **putBatch** the fully-derived `FootprintRecord[]` in **one IndexedDB transaction**
   (atomic; native `['hash','year','month']` upsert per row → idempotent).

**Recommendation flagged for PM ruling:** the literal "per row: convert→derive→put" loop
would leave a **partial commit** if row N's rate is uncached (rows 1..N-1 already written).
I recommend the **two-phase pre-flight** above so a rate gap = zero writes (all-or-nothing),
plus a new `FootprintDao.putBatch(records)` (single-transaction) for write-phase atomicity.
This is a small robustness upgrade over the literal loop; idempotency makes either safe on
retry, but two-phase avoids surprising partial states. **Confirm two-phase, or rule the
simple loop.**

## 3. The deferred proofs — real tests (the EP-3 payoff)

**Synthetic fixtures (always-run, CI):**
- **zero-growth re-import:** commit a statement, re-commit the SAME → `getAll().length`
  UNCHANGED (native idempotent upsert).
- **overlap:** commit Jan–Mar, then Jan–Dec (overlapping) → overlapping ops not
  double-counted (count = union, not sum).
- **in-month true-duplicate (FEAT-008 precision):** 2 byte-identical ops → **TWO** records
  (the 3.2 dup-counter gives them distinct hashes → distinct keys → both kept, NOT merged;
  count reconciles with the bank).
- **pseudo-op trio:** main + commission + cashback from one source row → **THREE** records
  (distinct `pseudoOp` discriminators → distinct hashes → distinct keys).

**Real statement (gated-skip, local-only):** a real **mono UA** statement under
`D:\abc-budget\*.csv` (LOCAL-ONLY, **never committed** — real financial data) run through
the full import pipeline → commit twice → **zero growth**; spot any genuine in-month dup.
**Guarded:** the test **skips** when the local fixture is absent (CI has no such file), so
it never blocks CI and never commits real data. Synthetic fixtures are the always-green proofs.

Rates for the proof dates are seeded into the cache (or `fetch` stubbed, per the 3.3 seam
harness) so `convert` succeeds — the proofs assert **counts/dedup**, not rate values.

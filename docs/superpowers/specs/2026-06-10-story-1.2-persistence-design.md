## Goal

Port the prior-art IndexedDB store module into the engine's internals and add the one thing it
lacks — a **versioned migration framework** — so stores can evolve safely across EP-3/4/6. Request
durable storage and expose a storage estimate **engine-internally**. Nothing leaks to the UI: the
public barrel and `EngineClient` are untouched, and the exports-resolution boundary test must stay
green unmodified.

## Prior-art assessment (verified by reading the source)

- `dao.ts` — `Dao<K,E>` + `BatchDao<K,E>` (CRUD, `list`/`find`/`findByIndex`/`getAllKeys`, batch
  create/read/update/upsert/delete). Clean, framework-free. **PORT verbatim.**
- `key.ts` — `SimpleKey` (string|number|Date) / `CompoundKey` + type guards. **PORT verbatim.**
- `idb/dao-impl.ts` (798 lines) — `IDBDao`/`IDBBatchDao`. IoC coupling is **shallow**: the container
  resolves exactly one thing (the `IDBDatabase`). **ADAPT:** constructor takes a
  `DbProvider = () => IDBDatabase` instead of `Container`; everything else verbatim.
- `idb/database.ts` — **migrations confirmed absent** (the PM's suspicion was right): store creation
  is idempotent-additive only, and the optional `migrationFn(db, old, new)` is a single synchronous
  monolithic callback that never receives the upgrade transaction — data transforms are effectively
  impossible. **SUPERSEDED** by the new migration framework; its `IDBStoreConfig`/`IDBIndexConfig`
  shapes survive as the `createStore`/`createIndex` argument types.
- `idb/index.ts` `registerDatabase` (IoC) — **DROPPED**; EP-2's facade port re-introduces IoC wiring
  around this module. The store stays pure.
- Specs (~1,900 lines, Jest-style, `fake-indexeddb@6`) — **ADAPT to vitest** (`jest.*`→`vi.*`;
  `@abc-budget/utils`' trivial `hasId` inlined; prior-art's `core-js` structured-clone polyfill
  dropped — Node 24 has native `structuredClone`).

## Hard constraints honored

- 🔒 **HC-2/3:** no raw-ledger assumptions — schema v1 creates **nothing** (no-op anchor); real

## 2. Migration framework (the new core)

```ts
interface MigrationStep {
  toVersion: number;                      // 1, 2, 3… contiguous from 1, validated at open
  migrate(ctx: MigrationContext): void;   // SYNC — IDB upgrade-tx constraint
}

interface MigrationContext {
  createStore(name, { keyPath?, autoIncrement?, indexes? }): IDBObjectStore;
  deleteStore(name): void;
  createIndex(storeName, indexName, keyPath, options?): void;
  deleteIndex(storeName, indexName): void;
  store(name): IDBObjectStore;            // raw access on the upgrade tx → data transforms
  rebuildStore(name, newOptions, transform?): void;  // keyPath-change escape hatch:
                                          // cursor-copy out → delete → recreate → re-insert (transformed)
}
```

`openDatabase(name, steps): Promise<IDBDatabase>`:

- Validates steps at call time: contiguous `toVersion` from 1, no duplicates — loud throw otherwise.
- DB version = last step's `toVersion`. `onupgradeneeded` replays steps `> oldVersion` in order,
  each receiving a ctx wrapping the native **version-change transaction**.
- **Promise semantics (explicit):** the returned promise **resolves only after the version-change
  transaction has committed** and the open request has succeeded — never mid-upgrade. If any step
  throws or the upgrade transaction aborts, the promise **rejects** and the database remains at
  `oldVersion`. There is no state in which a caller holds a DB whose migration is still in flight
  or partially applied.
- **Atomic by construction:** any throw aborts the upgrade transaction → *no* partial steps persist
  (an IndexedDB guarantee — the whole version change rolls back).
- **Sync discipline (the classic IDB pitfall, documented loudly):** `migrate()` must not await
  external promises — the upgrade tx auto-commits when its request queue drains. Helpers enqueue
  IDB requests on the tx, which is the one legal async pattern inside a migration.
- `onblocked` (another tab holds an older version): reject with a clear error in 1.2; multi-tab UX
  is a carry-forward.
- Schema is **defined by the replay** — there is no separate declarative schema to drift.

## 3. Proof suite (acceptance: "a version bump runs a migration cleanly on existing data")

Synthetic multi-step scenario against `fake-indexeddb@6` (devDep):

1. **Upgrade with data:** open v1 (create store, seed rows) → close → reopen v2 (add index) →
   v3 (data transform via `ctx.store()`) → v4 (`rebuildStore` with keyPath change + row transform) —
   data asserted correct at each stage.
2. **fresh≡upgrade invariant:** schema dump (storeNames + per-store keyPath/autoIncrement/indexes)
   of a fresh-at-v4 DB ≡ a stepwise-upgraded v1→v4 DB.
3. **Abort atomicity:** a step that throws mid-sequence → open rejects, DB still at old version,
   prior steps' effects absent.
4. **Validation:** gapped/duplicated `toVersion` → loud throw.

Plus the **ported DAO suite** (CRUD, compound keys, indexes, batch ops, error paths) and
**durability tests** (`vi.stubGlobal`-mocked `navigator.storage`: granted / denied / absent).

## 4. Production wiring + durability

`createDirectEngineClient()` fire-and-forgets an internal `initEnginePersistence()`:

- Lazily opens `abc-budget` at v1 (the **no-op anchor** — proves open/close + the migration plumbing
  on a real, empty DB; the version sequence is anchored for EP-3+ steps).
- Calls `requestDurability()`: `navigator.storage.persist()` + `estimate()`, results cached
  engine-internally. Guarded no-throw when the APIs are absent (node, worker spike).
- In 1.2, a real failure logs `console.warn` and is non-fatal — nothing reads the DB yet and no UI
  surface exists. **Carry-forward (EP-3, fail-loud):** the moment real user data lands (footprints),
  a `persist()` denial and any DB-open failure must surface **loudly** through the engine to the UI
  (FEAT-022 fail-loud posture), not warn-and-continue. Tagged in §6.
- Per RISK-003: `persist()` is best-effort (browsers may deny silently; we record the boolean). The
  real safety net is **user export** — a later epic, explicitly not this story.
- The worker entry (`engine.worker.ts`) does **not** init persistence — the real worker host is EP-2.6.

## 6. Out of scope / carry-forward

- Real stores arrive as migration steps v2+ with their epics: EP-3 footprints, EP-4 rules, EP-6 budget.
- Settings store → EP-2 (first real consumer: ENT-016 thresholds / base currency). The 1.4 language
  override deliberately uses `localStorage` (needs a sync read before first render), not this module.
- IoC wiring around the store → EP-2 facade port.
- **Fail-loud on `persist()` denial / DB-open failure → EP-3** (when footprints land; see §4).
- Cross-DAO transactions / unit-of-work → EP-2 facade concern.
- Multi-tab `onblocked` / `versionchange` handling beyond a clear rejection → carry-forward.
- **From the final review (latent traps for future step authors, all documented in JSDoc):**
  `rebuildStore` is in-line-keyed-stores-only (out-of-line keys throw; autoIncrement silently
  re-numbers — extend with `getAllKeys()` pairing if EP-3+ needs it); work chained inside a
  plain (non-cursor) request's success handler is not awaited by the step sequencer (cursor
  walks are — use them for read-then-write chains); `openEngineDb()` memoizes a rejected
  promise — add clear-on-reject before EP-2 wires real DAO consumers. Smaller: rebuild's
  re-insert `put()`s ride tx-FIFO rather than being tracked; durability tests don't cover the
  throwing-`persist()` branch.
- OPFS+SQLite escalation → YAGNI (ENT-007). User export/backup → FEAT-018, later epic.
- `getStorageInfo()` on `EngineClient` → declined for 1.2; revisit when the Settings UI needs it.

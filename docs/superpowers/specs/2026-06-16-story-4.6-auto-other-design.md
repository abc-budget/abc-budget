## 🔱 DECISION A — the transient-assignment REPRESENTATION  [the correctness fork]
**Confirmed:** a dumped row = `row.category = chosen` + **`isManuallySetCategory = false`** + no rule + no
override-map touch.
**The catch I verified:** `resolveCategory` returns `row.category` ONLY at L1, which requires
`isManuallySetCategory === true`. A dumped row (flag false) is therefore INVISIBLE to `resolveCategory` — it
still returns the rule result or `null`. So the dump cannot be expressed *through* `resolveCategory`; it is a
**lowest-precedence fallback BELOW the rules**:
```ts
effectiveCategory(row, ctx): Category | null  =  resolveCategory(row, ctx) ?? row.category
```
- **remainder** = rows where `effectiveCategory === null` (not resolved by L1/L2/L3 AND not dumped).
- **`isComplete`** = no row has `effectiveCategory === null`.
- **Distinct from a rule-categorized row:** a rule row has a MATCHING rule → `resolveCategory` returns its
  category (and `effectiveCategory` = that) → on re-import the rule re-applies → still categorized. A dumped
  row has NO matching rule and `isManual = 0` → on re-import `resolveCategory` is `null`, `row.category` is
  fresh-`null` (new import) → `effectiveCategory` = `null` → **uncategorized again** (the dump did NOT stick).
  vs a manual override (L1/L2, `isManual = 1`, sticky via the override map). **The re-import divergence is
  structural — no extra bookkeeping.**
- **Precedence — rules BEAT the dump** (because `resolveCategory ?? fallback`): correct — the dump is the
  last-resort "everything else"; if a rule later categorizes a dumped op properly, the rule wins.
- **Sandbox-edit-after-dump (flag, don't over-build):** after a dump, a 4.5 sandbox rule edit that newly
  matches a dumped row → `resolveCategory` now returns the rule category → `effectiveCategory` = the rule
  category (the dump is reclaimed). So the remainder/dump set can SHRINK after a rule edit. 4.6 only provides
  `effectiveCategory`; the recompute (re-run the gate) is the caller's (4.9) — not built here.
- **Commit seam (EP-5, out):** `deriveFootprint(row, usd, effectiveCategory(row)?.id ?? null, isManual)`
  where `isManual = 1` only when the win was L1/L2 (a dump is L-fallback → `isManual = 0`). 4.6 proves the
  round-trip directly (persist a dumped footprint `isManual:0` → re-resolve → `null`).

## 🔱 DECISION B — the magnitude SERVICE wiring  [fail-soft]
`computeRemainderMagnitude(remainderRows, { ratesService, base }): Promise<{ opCount; totalOpCount;
baseTotal; uncachedTail: Map<currency, number> }>`:
- The service = `ExchangeRateServiceImpl` over **`CacheOnlyRatesApi`** (offline, no network) + `base` from
  `getBaseCurrencyOrNull`. (If base unset → degrade: everything to the tail, or `baseTotal = null` — fail-soft.)
- Per remainder row: if `row.currency === base` → add `row.amount` to `baseTotal` directly (no conversion).
  Else `await ratesService.convert(row.amount, row.currency, base, row.date)`; on `RatesUnavailableError`
  (uncached) → **bucket `row.amount` into `uncachedTail.get(row.currency)`** (do NOT fail the dialog —
  fail-SOFT, unlike the commit's fail-loud).
- **Rate DATE = the OPERATION date `row.date`** — consistent with `deriveFootprint`/`footprint-usd` (UTC
  operation-date keying) and what `CacheOnlyRatesApi` reads. **Be explicit:** `warmRates` is EP-5, so
  pre-commit the operation-date rates are often UNcached → expect more rows in the tail at S3c; that is the
  designed best-effort signal («≈ N UAH + 320 ₴ uncached»), not a failure. Same-currency and any
  already-cached date convert exactly.
- **The number IS the signal — NO hard threshold, NO gate** (ENT-021: inform, don't gate). Magnitude never
  blocks «Решту → Інше».

## 2. DTO + (de)serialize
`ComplexRuleDTO { id?: number; rules: RuleDTO[]; categoryId: string; order: number }`,
`RuleDTO { field: string; operation: RuleOperation }`. Domain↔DTO gap bridged:
- **write:** `serializeComplexRule(complexRule, order)` → strip `evaluate()`, map `category.id → categoryId`
  (throws `LocalizableException` if the category has no id), `rules.map(serializeRule)`, attach `order`.
- **read:** `deserializeRule(dto)` rebuilds each `Rule`. The field→factory dispatch is ported, BUT:

### ADAPT — the private `amount` factory (minimal 4.1 touch, flagged)
4.1 made the bare amount factory **module-private** (only `createAmountCondition(op, currency)` is exported,
returning a `[amountRule, currencyEquals]` PAIR). A persisted amount condition was already stored as **two
separate RuleDTOs** (amount + currency=equals) at create time, so reload deserializes each independently —
no re-pairing needed. To rebuild the standalone `amount` Rule I will add an **additive** export to
`rule-factories.ts`: `rehydrateRule(field, operation): Rule | null` — the single field→factory dispatch,
living WITH the factories so it can reach the private amount builder. `deserializeRule` delegates to it.
(No re-export of the private factory; construction still goes through `createAmountCondition`.) **Flag:**
this is the one touch to a 4.1 file — additive, no behavior change to existing exports.

## 3. `complex-rules-dao.ts`
`ComplexRuleDAO extends IDBBatchDao<number, ComplexRuleDTO>` on `'complexRules'`, keyPath `'id'`. CRUD +
`getAllOrdered()` (sort by `order`) + `getByCategoryId(categoryId: string)` (via the `categoryId` index).
The DAO stores/returns **DTOs** (the domain↔DTO mapping lives in serialize/deserialize, invoked by the
service). Direct DI (`dbProvider`), no IoC.

## 4. `rule-persistence-service.ts` (RxJS-FREE)
Constructor: `(complexRuleDao, categoriesService)` — direct DI. (Uses the categories **service**, not the
raw DAO, so a rehydrated category's «base» currency is resolved at read — 4.3a.)
- `create(complexRule)` → next order = max(existing)+1; `dao.create(serialize(rule, order))`.
- `update(complexRule)` → preserve the existing `order`; `dao.update`.
- `reorder(ruleIds)` → read current `id→order`, compute new orders via 4.2 `reorderRuleOrders`, `dao`
  batch-write. **Only `order` changes; every id preserved.**
- **`reload(): Promise<DecisionTree>` — SYNCHRONOUS re-read (no RxJS):** `getAllOrdered()` → for each DTO,
  rehydrate `rules` via `rehydrateRule` and the `category` via `categoriesService.get(categoryId)`; build a
  fresh `DecisionTreeImpl` whose `complexRules` are **in `order` sequence** (array position = eval order →
  the 4.2 first-match evaluator honors order with **NO `order` field on `ComplexRule`** — see the flag).
- **`saveDecisionTree(tree)` — delta-ID-preservation (4.5 reuses):** ensure each category has an id (create
  via categories service if new); list existing DTOs; for each tree rule, content-match an unused existing
  DTO (same `categoryId` + same `rules` field/operation set) → **preserve its id**; unmatched tree rules →
  create (fresh autoInc id); existing DTOs not matched → **delete**. Order assigned by tree index. Ported
  RxJS-free.

## 🔱 RULING (PM-required) — dangling `categoryId` on reload
A complexRule whose `categoryId` resolves to **no category**. Note 4.3a has **no hard-delete** (archive
only), and the categories DAO `read(id)`/service `get(id)` **return archived categories** — so a rule
pointing at an *archived* category still rehydrates fine (its category just carries `isArchived: true`).
A truly dangling id is only reachable via data corruption or a future hard-delete.
**Ruling (recommend): SKIP the dangling rule from the reloaded tree + log loudly (`logger.error`), and
RETAIN it in the store (do NOT delete).** Rationale: fail-soft (never silently corrupt the tree or
mis-categorize), but non-destructive — if the category reappears (un-archive / re-create / restore), the
rule loads again. Deleting would be irreversible data loss on a transient gap. **Confirm.**
*Alt rejected:* a placeholder category (pollutes the model + risks a wrong-category apply).

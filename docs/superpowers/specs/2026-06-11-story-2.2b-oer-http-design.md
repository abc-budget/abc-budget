## Dev verification (work item 2) — RESULT: same-origin GO

Current Firebase docs (checked 2026-06-11): Hosting rewrites support an explicit `region` for both
2nd-gen functions (`"function": { "functionId", "region", "pinTag" }`) and Cloud Run
(`"run": { "serviceId", "region" }`); **europe-west1 is in the supported region list** (the region
field merely defaults to us-central1 when omitted; multi-region functions make it mandatory).
→ **`/api/rates` Hosting rewrite + relative-path fetch + reject ALL cross-origin.**

**Design consequence (Origin guarantee):** the request stays **POST** (JSON `{date}`) — browsers
send `Origin` on ALL POSTs including same-origin; plain GETs may omit it, which would weaken the
hard-reject. POST also mirrors the onCall wire shape (minimal client diff).

## Request-acceptance policy (the precise rule)

Accept only when ALL hold, in order, before any I/O:
1. `POST` + JSON body with valid date (format + 1999-01-01‥today — existing validators).
2. **Origin hard-reject:** `Origin` present AND ∈ allowlist `[https://abc-budget-2d379.web.app,
   https://abc-budget-2d379.firebaseapp.com]` — NO localhost in prod (any local process can claim
   it; ENT-004). Secondary signal: if `Sec-Fetch-Site` is present it must be `same-origin`/`same-site`
   (forbidden-header class — browsers can't fake it; absent = older agent, Origin alone decides).
3. **Per-IP rate limit:** in-instance token bucket (e.g. 10 req/min/IP, burst 10) → 429. Light,
   best-effort (per-instance; maxInstances bounds the multiplier).
Then the existing pipeline: Firestore by-date cache → budget cap (transactional, cache-hits free) →
OER → save → return. `maxInstances: 2` on the function. Errors structured, zero internals.
Spoofability is DOCUMENTED, not fought: curl can fake Origin — layers 1/5/6/7 of ENT-004 hold the
quota/billing regardless.
